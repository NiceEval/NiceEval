In niceeval, `Agent` is the abstraction the core calls and `Adapter` is the concrete implementation. This page answers one central question: **under Tier 1 (non-invasive) integration, why does an Adapter only need a single `send` function to support the entire drive API on the eval side — `t.send()`, `t.sendFile()`, `t.newSession()`, and HITL's `t.respond()` / `t.respondAll()`.**

## Two terms, one boundary

- **Agent** is the subject under test as the core sees it: a named object with capability flags and a single `send` method. The runner interacts with nothing but this interface — it never branches on `if (agent === "claude-code")`.
- **Adapter** is the code implementing that interface. You write it (or use a built-in). It knows how to authenticate, how to call your application, and how to translate whatever comes back into the standard event stream.

The URL, credentials, and protocol details of the system under test are the Adapter's private configuration (read from environment variables or closure) — the core never sees them. Experiments reference Agents by name; positional arguments after the experiment name only select *which evals to run*.

```bash
# Same evals, different Adapter; the URL is the Adapter's private business
npx niceeval exp local support
npx niceeval exp prod support
```

## The Agent contract: one verb, one product

Whether the Adapter talks to an HTTP service or a CLI in a sandbox, the interface exposed to the runner is identical:

```ts
interface Agent {
  readonly name: string;
  readonly capabilities: AgentCapabilities;
  send(input: TurnInput, ctx: AgentContext): Promise<Turn>;
  // Optional: setup / teardown (once-per-sandbox lifecycle), tracing (OTLP export config),
  // spanMapper (native span → canonical normalization), events (otelEvents() declares
  // that events derive from spans)
}

interface TurnInput {
  readonly text: string;
  readonly files?: readonly InputFile[];   // files attached by t.sendFile (base64)
}

interface Turn {
  readonly events: StreamEvent[];          // ★ standard event stream — every Adapter's core product
  readonly data?: unknown;                 // structured output (for outputEquals / outputMatches)
  readonly status: "completed" | "failed" | "waiting";  // waiting = parked on human input (HITL)
  readonly usage?: Usage;
}
```

`send` is the **single verb**. `Turn.events` is the **single product**. The entire difference between Adapters lives in how `send` translates the raw response into `events`.

## Tier 1: what "non-invasive" means

**Not a single line of application code changes.** The application is started by you, in its own way (`pnpm start`, deployed anywhere); the eval side does not spawn the application process and does not open extra ports — the Adapter only talks to the application's **existing external interface** (HTTP endpoints, SSE streams), and reports a clear "start the app first" error when it can't connect. Observability comes from what the interface already returns, or from OTel (currently documented in the Chinese guide 通过 OTel 接入).

This makes the Adapter the **only contact surface** between the eval and the application. Everything below builds on one fact: the five drive APIs on the eval side, however different they look, all converge into "call `send` once more" by the time they reach the Adapter — the only difference is the state of `TurnInput` and `ctx.session`.

## Five t APIs, three cases inside send

| Eval side | At the Adapter | What `send` receives |
|---|---|---|
| `t.send(text)` | one `send` call | `input.text`; first turn on this session line has `ctx.session.isNew === true` |
| `t.sendFile(path, text?)` | one `send` call | same, plus `input.files` (base64 `InputFile[]`); ignore it if your app takes no files |
| `t.newSession()` | no `send` by itself | opens a second session line; that line's next `send` gets `ctx.session = { isNew: true, id: undefined }` |
| `t.respond(...answers)` | one **ordinary** `send` | `input.text` = the answers joined with `\n`; same session line |
| `t.respondAll(optionId)` | one **ordinary** `send` | same, one line of `optionId` per pending request |

So an Adapter author does not implement five methods. The real obligation is to make **one** `send` recognize three cases:

1. **A fresh turn**: `ctx.session.isNew === true` and nothing held → start a new conversation with the app.
2. **A continuing turn**: `isNew === false` → continue the same history (*how* to continue depends on your app's interface shape — see question two below).
3. **Resuming a parked turn**: the previous turn ended with `status: "waiting"` → this turn's `input.text` is the human verdict; deliver it to the app first, then keep reading the stream you left unfinished.

Case 3 also requires the *previous* turn to have met HITL's two obligations: when the agent stops to wait for a human, `send` returns `status: "waiting"` and emits one `input.requested` event per pending question (that is what `t.parked()` and `t.requireInputRequest()` read).

## Three questions to answer before writing send

A hand-written send really contains three unrelated jobs: **transport** (how to make the request — genuinely irreducible and not worth abstracting, since every backend's URL, auth, and body shape differ; this is the only segment you hand-write), **reduce** (raw response → `StreamEvent[]`), and **orchestration** (session continuation + HITL pause/resume). The latter two ship as official pieces (exported from `niceeval/adapter`), so writing send is not "implement from scratch" — it is answering three questions and composing the pieces those answers point to.

### Question one: how does the app's response arrive? — picks your parser

| Response shape | Use | You write |
|---|---|---|
| Relays an SDK's native event stream (one frame = one complete unit) | `fromClaudeSdkMessages()` / `fromPiAgentEvents()` / `fromCodexThreadEvents()` | zero mapping |
| A full AI SDK `generateText` / `streamText` result | `fromAiSdk(result)` | zero mapping |
| Token-by-token / argument-delta stream | the protocol's own official reducer; else `deltaStream({ toOps })` | one "frame type → operation" mapping table |
| Interface returns only the final answer, but the app is OTel-instrumented | `events: otelEvents()` — events derive from spans | zero mapping (send is transport only) |
| Interface returns only the final answer, no process data at all | hand-write a few events | start with one `message`; don't declare capabilities you can't deliver |

The preset parsers go as far as **shape**, not protocol: niceeval defines no protocol, so a universal parser cannot exist — but there are only this many arrival shapes, and each has one official piece. The only case where you write a mapping is "delta stream with no official reducer from the protocol's side" — and even then you write a declaration (which operation this frame maps to); buffering, pairing, and flush timing are handled by `deltaStream`, not a hand-rolled state machine.

### Question two: how does turn two continue turn one? — picks the session branch in send

Figure out which kind your app's interface is. There are exactly two modes in the wild, each with a symmetrically shaped official piece (one strategy object, one pair of methods):

- **Server keeps the history** (the interface takes a session id: native SDK sessions / threads, the OpenAI Responses API) → `serverSession()`. Send requests with `session.id(ctx)`; when the response reveals an id, write it back with `session.capture(ctx, id)`.
- **Client carries the full history** (stateless server, every turn posts the complete message list: Chat-Completions-shaped interfaces) → `clientHistory<TMsg>()`. `get(ctx)` retrieves this line's history, append this turn's message, `commit(ctx, messages)` writes the updated list back.

Once this question is answered, "first send / second send / `t.newSession()`" need **no separate handling**: on `isNew`, `session.id(ctx)` returns `undefined` and `clientHistory.get` returns an empty array — a fresh session line automatically falls back to the "first send" shape, and the next turn on the same line automatically carries the id or the history. `capture` also guards a subtle trap for you: it only lands on a fresh line's first turn, so a resume turn is never clobbered by a (possibly forked) new id from the backend.

### Question three: does it stop mid-turn to wait for a human? — decides whether you need a HITL branch

- **Yes** (tool approval, missing information): `pausable<TState>()` plus `driveFrameStream`'s `{ pause }` return value. On parking, `hold(ctx, state)` records the half-read stream; the next send calls `take(ctx)` first, delivers the verdict to the app, then keeps reading the same stream. `driveFrameStream` appends the `input.requested` event and sets status to `"waiting"` for you.
- **No**: skip the branch. `send` never returns `"waiting"`, and `t.respond` never appears in your evals.

<Tip>
**OTel answers question one only — never questions two and three**: spans have no "waiting for input" semantics and don't know whether this turn should carry history. Whether events come from frames or from spans, session continuation and HITL are answered separately by the two questions above.
</Tip>

## Putting it together: one non-invasive Adapter supporting all five APIs

Below is the full skeleton of a Tier 1 Adapter (against an already-running app that relays its native event stream as SSE; complete runnable versions in the [tier1 examples](https://github.com/CorrectRoadH/niceeval/tree/main/examples/zh/tier1)). Its answers to the three questions: **relayed pi native event stream (Q1) → `fromPiAgentEvents`; server keeps the history (Q2) → carry a session id; parks for approval (Q3) → `pausable`**. Note which line supports which t API:

```ts
// agents/my-app.ts
import { defineAgent, sseJsonFrames, fromPiAgentEvents, driveFrameStream, pausable, serverSession } from "niceeval/adapter";
import type { AgentContext, PiAgentStream, SseFrameCursor } from "niceeval/adapter";
import type { Turn, TurnInput } from "niceeval";

const BASE_URL = process.env.MY_APP_URL ?? "http://127.0.0.1:5299";  // the app's own port; evals don't manage its process

interface Pending { cursor: SseFrameCursor<Frame>; stream: PiAgentStream; callId: string }
const pending = pausable<Pending>();                    // ← Q3; supports t.respond / t.respondAll
const session = serverSession();                        // ← Q2; supports t.newSession / multi-turn resume

function readStream(cursor: SseFrameCursor<Frame>, ctx: AgentContext, stream: PiAgentStream): Promise<Turn> {
  return driveFrameStream(cursor, stream, ctx, (frame) => {
    if (frame.type === "session") {
      session.capture(ctx, frame.sessionId);            // ← Q2: write the returned id back for resume
      return;
    }
    if (frame.type === "approval_request") {            // ← Q3: the app stopped to wait for approval
      pending.hold(ctx, { cursor, stream, callId: frame.toolCallId });
      return { pause: { id: frame.toolCallId, action: frame.toolName,
                        options: [{ id: "approve" }, { id: "deny" }] } };
      // On pause, driveFrameStream appends an input.requested event, sets status to
      // "waiting", and stops reading the stream
    }
    if (frame.type === "server_error") return { fail: frame.message };
  });
}

async function send(input: TurnInput, ctx: AgentContext): Promise<Turn> {
  const held = pending.take(ctx);
  if (held) {                                           // ← t.respond("approve"): ordinary send; deliver the verdict first
    const approved = input.text.trim().toLowerCase() === "approve";
    if (!approved) held.stream.markRejected(held.callId);
    await fetch(`${BASE_URL}/api/chat/approve`, { method: "POST", signal: ctx.signal,
      body: JSON.stringify({ toolUseId: held.callId, approved }) });
    return readStream(held.cursor, ctx, held.stream);   //    then keep reading the same stream — no new request
  }

  const res = await fetch(`${BASE_URL}/api/chat`, {     // ← transport: the only truly hand-written segment
    method: "POST",
    body: JSON.stringify({
      message: input.text,                              // ← t.send
      sessionId: session.id(ctx),                       // ← Q2: first send automatically carries none, later turns do
    }),
    signal: ctx.signal,
  });
  return readStream(sseJsonFrames<Frame>(res.body!), ctx, fromPiAgentEvents());  // ← Q1: official converter
}

export default defineAgent({
  name: "my-app",
  capabilities: { conversation: true, toolObservability: true },
  send,
});
```

Change an answer and you only swap the corresponding piece: a Chat-Completions-shaped interface (Q2 → client carries the full history) replaces `serverSession` with `clientHistory`'s `get`/`commit`; no HITL (Q3 → no) deletes all three `pending` spots. The few `fetch` lines and the `onFrame` judgment of "does this frame need extra handling" are the author's only hand-written parts.

`t.sendFile` in this skeleton is just one more read of `input.files`: put the base64 files into the request body however your app's interface expects; if the interface takes no files, ignore them.

## Capability flags shape your `t` context

Capability flags are the Adapter's promise to eval authors. They control which `t` methods are visible at the TypeScript level (with runtime guards as well):

| Capability | Methods unlocked on `t` |
|---|---|
| *(any Agent)* | `t.send`, `t.sendFile`, `t.check`, `t.require`, `t.judge`, `t.log`, `t.skip` |
| `conversation` | multiple `t.send()` calls, `t.reply`, `t.newSession()`, and HITL's `t.respond()` / `t.respondAll()` / `t.requireInputRequest()` |
| `toolObservability` | the full scoped assertion set: `t.calledTool()` / `t.notCalledTool()` / `t.toolOrder()` / `t.usedNoTools()` / `t.maxToolCalls()` / `t.noFailedActions()` / `t.event()` … |
| `workspace` / `sandbox` | `t.sandbox`, `t.sandbox.diff`, `t.fileChanged()` … (sandbox Agents) |
| `compactionObservability` | context-compaction assertions |
| `tracing` | `EvalResult.trace`, the `niceeval view` waterfall (once declared, the runner provides a `ctx.telemetry` receiver endpoint) |

The rule: **when unsure, don't declare.** Declaring a capability you don't deliver makes negative assertions (`notCalledTool` and friends) silently unreliable. Event streams produced by official converters carry their own completeness proof; hand-written mappings are on you. Per-capability obligations: [defineAgent reference](/reference/define-agent).

## The standard event stream

The most important thing an Adapter does is not connecting to the AI — it is **normalizing the AI's output into the standard event stream**. Once that happens, the entire assertion vocabulary works for free, regardless of which AI produced the data:

```ts
type StreamEvent =
  | { type: "message"; role: "assistant" | "user"; text: string }
  | { type: "action.called"; callId: string; name: string; input: JsonValue; tool?: ToolName }
  | { type: "action.result"; callId: string; output?: JsonValue;
      status: "completed" | "failed" | "rejected" }
  | { type: "subagent.called"; callId: string; name: string; remoteUrl?: string }
  | { type: "subagent.completed"; callId: string; output?: JsonValue;
      status: "completed" | "failed" }
  | { type: "input.requested"; request: InputRequest }
  | { type: "thinking"; text: string }
  | { type: "compaction"; reason?: string }
  | { type: "error"; message: string };
```

The core's `deriveRunFacts(events)` folds this flat stream into structured facts (`toolCalls`, `subagentCalls`, `parked`, `messageCount`, …) and all scoped assertions read from those derived facts. Your Adapter only needs to produce correct events; scoring is handled entirely by the core.

## The other transport: sandbox Agents

To evaluate file-editing coding agent CLIs (claude-code / codex / bub), use `defineSandboxAgent`: the contract is identical (same `send`, same `Turn`); the only difference is that the "connection" is not HTTP but "spawn the CLI inside a sandbox and read back the transcript", with the runner passing the sandbox handle via `ctx.sandbox`. The three built-in Agents work out of the box — see the [sandbox agent guide](/guides/sandbox-agent) for using them and wiring up a new CLI.

## Agent × Sandbox orthogonality

`experiment.agent` selects who performs the task; `experiment.sandbox` (`dockerSandbox()` / `vercelSandbox()` / `e2bSandbox()`) selects where; the `sandbox` in `niceeval.config.ts` is the project-wide fallback (`exp.sandbox ?? config.sandbox`). Any sandbox Agent works with any sandbox backend; Tier 1 non-invasive Adapters never touch a sandbox at all. There is no CLI flag and no auto-detection — a sandbox Agent's experiment (or the config fallback) must set `sandbox` explicitly, or niceeval throws when creating one.
