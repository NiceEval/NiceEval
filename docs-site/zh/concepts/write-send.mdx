[Agent 与 Adapter](/zh/concepts/agents-adapters) 讲了契约：`send` 传入 `TurnInput` 和 `AgentContext`，返回 `Turn`。这一页讲怎么写它——先看清 eval 侧的五个驱动 API 到 send 只剩三种情况，再回答三个问题、把答案对应的官方件拼起来，最后用两个完整 Adapter 和一个流式 + HITL 骨架把链路走通。

## 五个 t API，到 send 只是三种情况

运行器在会话上只承诺一件事：**同一条会话线的每次 `send` 拿到同一份 `ctx.session.state`，新会话线拿到一份空的。**这份状态从 `{}` 起步，写什么、什么时候写完全是 Adapter 的事——框架从不往里放数据，也没有任何要你检查的标志位。

| eval 侧 | 到 Adapter 时 | `send` 收到什么 |
|---|---|---|
| `t.send(text)` | 一次 `send` | `input.text`；`ctx.session.state` 是这条会话线的自有状态 |
| `t.sendFile(path, text?)` | 一次 `send` | 同上，外加 `input.files`（base64 的 `InputFile[]`）；不支持多模态就忽略 |
| `t.newSession()` | 不直接触发 `send` | 开第二条会话线；该线随后第一次 `send` 拿到一份**全空的** `state`——和整个 eval 的第一次 `send` 形态完全相同 |
| `t.respond(...answers)` | 一次**普通的** `send` | `input.text` = 回答文本；`input.responses` 逐请求一条 `{ requestId, optionId }`（自由文本回答则是 `{ requestId, text }`，见[不同回答的入参](/zh/concepts/agents-adapters#不同回答的入参)）；还是同一条线、同一份 `state` |
| `t.respondAll(optionId)` | 一次**普通的** `send` | 同上，每条待处理请求各一条回答；`optionId` 已在 eval 侧对每条请求校验过存在，写错直接抛，不会静默传给应用 |

所以 Adapter 作者不需要实现五个方法，也没有框架状态要判断——`send` 里你只读自己存过的东西，状态的三种内容对应三种情况：

1. **状态是空的** → 对应用发起新会话。eval 的第一轮和 `t.newSession()` 后的第一轮形态完全相同，不需要区分。
2. **状态里有你存的会话 id / 历史** → 接着同一条历史（怎么"接"取决于应用的接口形状，见下面问题二）。
3. **状态里有你挂起的半途流** → 上一轮以 `status: "waiting"` 停下；先把 `input.responses` 里的人工裁决交回应用，再接着读上次没读完的流。

这三种差异来自应用接口的真实差异（带不带 id、发不发历史、怎么交回裁决），不是框架强加的协议——框架的全部介入就是开头那一句承诺。第 3 种情况还要求"上一轮"先尽到 HITL 的两条义务：agent 停下等人时，`send` 返回 `status: "waiting"`，并且每个待回答的问题吐一条带稳定 `id` 的 `input.requested` 事件（`t.parked()`、`t.requireInputRequest()` 读它，回答靠这个 `id` 对位）。

## 写 send 之前，先回答三个问题

一个手写 send 里其实只有三件互不相干的事：**transport**（怎么发——真做不掉也不需要抽象，每家的 URL、鉴权、请求体本来就不一样，这是你唯一手写的一段）、**reduce**（原始返回 → `StreamEvent[]`）、**编排**（会话续接 + HITL 暂停恢复）。后两件都有官方件（从 `niceeval/adapter` 导出），所以写 send 不是"从零实现"，而是先回答三个问题，再把答案对应的件拼起来。这些件的存取都落在上一节那份会话线状态上——用了它们，这份状态你连碰都不用碰。

### 问题一：接口是流式的吗？——决定 parser 怎么选

接的位置永远是**用户前端本来就在用的那个接口**：页面调哪个端点、收什么格式，Adapter 就调哪个端点、收什么格式——不为 eval 开新接口，也不从应用内部拿数据。所以这个问题只看接口本身：**它是流式的，还是一问一答？**

**流式**（SSE / NDJSON，前端边收边渲染）——再看流里跑的是什么：

| 流里是什么 | 用什么 | 你写什么 |
|---|---|---|
| SDK 原生事件透传（一帧一个完整单元） | `fromClaudeSdkMessages()` / `fromPiAgentEvents()` / `fromCodexThreadEvents()` | 零映射 |
| AI SDK 的 UI Message Stream（`useChat` 协议） | `uiMessageStreamAgent()` | 零映射（整个 Agent 都是现成的，连 send 都不用写） |
| 逐 token / 逐参数增量 | 协议方官方 reducer；没有才用 `deltaStream({ toOps })` | 一张"帧类型 → 操作"映射表 |

**非流式**（一问一答，一个 JSON 回来）——再看返回里有什么：

| 返回里有什么 | 用什么 | 你写什么 |
|---|---|---|
| Chat Completions 形返回 | `fromChatCompletion(res)` | 零映射 |
| Responses 形返回 | `fromResponses(res)` | 零映射 |
| AI SDK `generateText` 的完整结果 | `fromAiSdk(result)` | 零映射 |
| 只有最终答案 | 手写几条 events | 一条 `message` 起步 |

预设的 parser 只到"**形状**"为止，不到"协议"为止：NiceEval 不定协议，所以不存在万能 parser；但接口的到达形状统共就这几种，每种一个官方件。只有"增量流且协议方没有官方 reducer"这一种情况需要你写映射——而且写的是声明（这一帧对应哪个操作），拼接、配对、落地时机由 `deltaStream` 统一管，不是手写状态机。

### 问题二：第二轮怎么接上第一轮？——决定 send 里的会话分支

先想清楚你的应用接口属于哪种。业界统共两种模式，各对应一个形状对称的官方件（一个策略对象、一对方法）：

- **服务端记历史**（接口收一个会话 id：各 SDK 的原生 session / thread、OpenAI Responses API 都是）→ `serverSession()`。发请求带 `session.id(ctx)`；从返回里抓到 id 就 `session.capture(ctx, id)` 写回。
- **客户端带全量历史**（服务端无状态，每轮发完整消息列表：Chat Completions 形接口是典型）→ `clientHistory<TMsg>()`。`get(ctx)` 取出本线历史、追加本轮消息发出去、`commit(ctx, messages)` 写回最新列表。

回答了这个问题，"第一次发 / 第二次发 / `t.newSession()`"就**不需要单独处理**：新会话线的状态是空的，`session.id(ctx)` 自然返回 `undefined`、`clientHistory.get` 自然返回空数组——"第一次发"的形态是空状态的自然结果，不是要你判断的分支；同一条线的下一轮自动带上 id 或历史。`capture` 只在还没记过 id 时落地，后端重复回传（甚至因 fork 变了）也不会覆盖正在续接的线。

### 问题三：中途会停下来等人吗？——决定要不要 HITL 分支

- **会**（工具审批、等补充信息）：`pausable<TState>()` + `driveFrameStream` 的 `{ pause }` 返回值。停轮时 `hold(ctx, state)` 记下读了一半的流；下一轮 send 先 `take(ctx)`，把裁决交给应用，再续读同一条流。`driveFrameStream` 会自动补 `input.requested` 事件、把 status 置 `"waiting"`。
- **不会**：跳过这一分支。send 永不返回 `"waiting"`，eval 里也不会出现 `t.respond`。

## 标准事件流

三个问题的产出汇到同一个地方。Adapter 最重要的工作不是"连上 AI"，而是**把接口的返回归一成标准事件流**——parser 件产出的是它，手写映射的转换目标也是它。所谓"流"，落到 `Turn.events` 上就是**一个普通数组**：本轮发生的每件事一个对象，按真实发生顺序排列，对象是下面九种类型之一（具体长什么样，[契约页有一轮的实际值](/zh/concepts/agents-adapters)）。归一完成后，整套断言词汇对任何 AI 免费可用：

```ts
type StreamEvent =
  | { type: "message"; role: "assistant" | "user"; text: string }
  | { type: "action.called"; callId: string; name: string; input: JsonValue; tool?: ToolName }
  | { type: "action.result"; callId: string; output?: JsonValue;
      status: "completed" | "failed" | "rejected" }
  | { type: "subagent.called"; callId: string; name: string; remoteUrl?: string }
  | { type: "subagent.completed"; callId: string; output?: JsonValue;
      status: "completed" | "failed" }
  | { type: "input.requested"; request: InputRequest }
  | { type: "thinking"; text: string }
  | { type: "compaction"; reason?: string }
  | { type: "error"; message: string };
```

这条流是断言词汇的**货架**：你吐哪种事件，eval 作者就能写哪族断言——

| 你吐的事件 | 解锁的断言 |
|---|---|
| `message` | `t.reply`、`t.messageIncludes()`、judge 的对话材料 |
| `action.called` / `action.result`（靠 `callId` 配对） | `t.calledTool()` / `t.toolOrder()` / `t.maxToolCalls()` / `t.noFailedActions()`… |
| `input.requested`（配合 status `"waiting"`） | `t.parked()`、`t.requireInputRequest()`、`t.respond()` |
| `thinking` / `compaction` / `error` | 对应断言与 o11y 计数 |

而事件**几乎从不手写**：问题一选中的转换器，返回值就是 `events`（连 `usage`、`status` 一起，拿来直接 `return`）。手写只发生在"接口只回最终答案"的兜底档。大多数接口用得上的也只有前两行：`message` 加 `action.called` / `action.result`。下面两章拿两种最熟悉的接口形状，把"接口返回 → 事件 → 断言"这条链路整个走一遍。

## 例一：Chat Completions 形接口

三个问题的答案：**非流式、Chat Completions 形（问题一）→ `fromChatCompletion`；客户端带全量历史（问题二）→ `clientHistory`；不停轮等人（问题三）→ 无 HITL 分支**。Adapter 全文：

```ts
// agents/chat-app.ts
import { defineAgent, clientHistory, fromChatCompletion } from "niceeval/adapter";
import type { AgentContext } from "niceeval/adapter";
import type { Turn, TurnInput } from "niceeval";

const BASE_URL = process.env.CHAT_APP_URL ?? "http://127.0.0.1:8080";

interface Msg { role: "user" | "assistant"; content: string | null }
const history = clientHistory<Msg>();               // ← 问题二：服务端无状态，历史由这里记

async function send(input: TurnInput, ctx: AgentContext): Promise<Turn> {
  const messages = [...history.get(ctx), { role: "user" as const, content: input.text }];

  const res = await fetch(`${BASE_URL}/v1/chat/completions`, {   // ← 前端在用的那个接口
    method: "POST", signal: ctx.signal,
    body: JSON.stringify({ messages }),                          // ← 每轮发完整历史
  }).then((r) => r.json());

  history.commit(ctx, [...messages, res.choices[0].message]);    // ← 写回；下一轮 get 自动带上

  return fromChatCompletion(res);   // ← 问题一：events + usage + status 都在返回值里
}

export default defineAgent({
  name: "chat-app",
  session: history,        // ← 把用的件递给定义：多轮能力由此而来，没有布尔声明
  send,
});
```

`fromChatCompletion` 内部没有魔法，展开就是"转换目标"那几种事件：

```ts
// fromChatCompletion(res) 等价于：
const reply = res.choices[0].message;
const events: StreamEvent[] = [];
for (const call of reply.tool_calls ?? []) {
  events.push({ type: "action.called", callId: call.id,
                name: call.function.name, input: JSON.parse(call.function.arguments) });
}
if (reply.content) events.push({ type: "message", role: "assistant", text: reply.content });
return { events, status: "completed",
         usage: { inputTokens: res.usage.prompt_tokens, outputTokens: res.usage.completion_tokens } };
```

会话上没有任何"第一轮"分支——新会话线（包括 `t.newSession()` 之后）`history.get` 自然返回空数组。一个如实交代：Chat Completions 这个形状**不承诺**"返回 = 全量过程记录"（应用完全可能在服务端跑完工具循环、只把最终答案吐给你），所以 `fromChatCompletion` 的返回不带完整性证明——`calledTool` 这类正断言照常可用，`notCalledTool` 这类负断言会提示不可信。这不是你配置错了，是这个接口形状能给到的上限。

## 例二：Responses API 形接口

三个问题里只有问题二的答案变了：**服务端记历史 → `serverSession` + `previous_response_id`**。和例一逐行对比着看：

```ts
// agents/responses-app.ts
import { defineAgent, serverSession, fromResponses } from "niceeval/adapter";
import type { AgentContext } from "niceeval/adapter";
import type { Turn, TurnInput } from "niceeval";

const BASE_URL = process.env.RESPONSES_APP_URL ?? "http://127.0.0.1:8080";

const session = serverSession();                    // ← 问题二：历史在服务端，这里只记 id

async function send(input: TurnInput, ctx: AgentContext): Promise<Turn> {
  const res = await fetch(`${BASE_URL}/v1/responses`, {          // ← 前端在用的那个接口
    method: "POST", signal: ctx.signal,
    body: JSON.stringify({
      input: input.text,
      previous_response_id: session.id(ctx),        // ← 新会话线自动不带，之后自动带
    }),
  }).then((r) => r.json());

  session.capture(ctx, res.id);                     // ← 回传的 response id 记下，下一轮续接

  return fromResponses(res);   // ← 问题一：遍历 output——function_call → action.called、message → message
}

export default defineAgent({
  name: "responses-app",
  session,                 // ← 同例一：件即能力
  send,
});
```

两章的差异清单，正好就是三个问题的答案差：transport 的参数从 `messages`（全量历史）换成 `previous_response_id`（一个 id）；会话件从 `clientHistory` 换成 `serverSession`；转换器从 `fromChatCompletion` 换成 `fromResponses`。还有一个例一给不了的东西：Responses 这个形状**承诺** `output` 数组是这次响应的全量过程记录（协议契约，不是应用自觉），所以 `fromResponses` 的返回自带完整性证明——`notCalledTool` 这类负断言在例二可信。同样零映射、同样一行声明都没写，可信度的差别完全来自接口形状本身。

## 再进一步：流式 + HITL 的完整骨架

例一例二都是"非流式、不等人"的最小形态。接口是流式的、而且会停轮等审批时，骨架长这样（对着一个已在跑、以 SSE 透传原生事件流的应用；完整可跑版本见 [tier1 示例](https://github.com/CorrectRoadH/niceeval/tree/main/examples/zh/tier1)）。它对三个问题的回答是：**流式接口、流里是透传的 pi 原生事件（问题一）→ `fromPiAgentEvents`；服务端记历史（问题二）→ 带会话 id；有审批停轮（问题三）→ `pausable`**。注意每一段分别在支撑哪个 t API：

```ts
// agents/my-app.ts
import { defineAgent, sseJsonFrames, fromPiAgentEvents, driveFrameStream, pausable, serverSession } from "niceeval/adapter";
import type { AgentContext, PiAgentStream, SseFrameCursor } from "niceeval/adapter";
import type { Turn, TurnInput } from "niceeval";

const BASE_URL = process.env.MY_APP_URL ?? "http://127.0.0.1:5299";  // 应用自己的端口，eval 不代管进程

interface Pending { cursor: SseFrameCursor<Frame>; stream: PiAgentStream; callId: string }
const pending = pausable<Pending>();                    // ← 问题三；支撑 t.respond / t.respondAll
const session = serverSession();                        // ← 问题二；支撑 t.newSession / 多轮续接

function readStream(cursor: SseFrameCursor<Frame>, ctx: AgentContext, stream: PiAgentStream): Promise<Turn> {
  return driveFrameStream(cursor, stream, ctx, (frame) => {
    if (frame.type === "session") {
      session.capture(ctx, frame.sessionId);            // ← 问题二：回传的 id 写回，下一轮靠它续接
      return;
    }
    if (frame.type === "approval_request") {            // ← 问题三：应用停下等审批
      pending.hold(ctx, { cursor, stream, callId: frame.toolCallId });
      return { pause: { id: frame.toolCallId, action: frame.toolName,
                        options: [{ id: "approve" }, { id: "deny" }] } };
      // driveFrameStream 收到 pause：补一条 input.requested、status 置 "waiting"、停止读流
    }
    if (frame.type === "server_error") return { fail: frame.message };
  });
}

async function send(input: TurnInput, ctx: AgentContext): Promise<Turn> {
  const held = pending.take(ctx);
  if (held) {                                           // ← t.respond("approve")：普通 send，先交裁决
    const approved = input.responses?.[0]?.optionId === "approve";  // 多请求并停时按 requestId 对位
    if (!approved) held.stream.markRejected(held.callId);
    await fetch(`${BASE_URL}/api/chat/approve`, { method: "POST", signal: ctx.signal,
      body: JSON.stringify({ toolUseId: held.callId, approved }) });
    return readStream(held.cursor, ctx, held.stream);   //    再接着读同一条流，不重发请求
  }

  const res = await fetch(`${BASE_URL}/api/chat`, {     // ← transport：唯一真正手写的一段
    method: "POST",
    body: JSON.stringify({
      message: input.text,                              // ← t.send
      sessionId: session.id(ctx),                       // ← 问题二：第一次发自动不带，之后自动带
    }),
    signal: ctx.signal,
  });
  return readStream(sseJsonFrames<Frame>(res.body!), ctx, fromPiAgentEvents());  // ← 问题一：官方转换器
}

export default defineAgent({
  name: "my-app",
  session,     // ← 件即能力；fromPiAgentEvents 透传 SDK 全量事件流，负断言的完整性证明也随之而来
  send,
});
```

换一组答案，换的只是对应的件：会话件怎么互换，例一例二已经逐行演示过；没有 HITL（问题三选"不会"）就删掉 `pending` 相关的全部三处。transport 那几行和 `onFrame` 里"这一帧要不要额外处理"的判断，是作者仅有的手写部分。

`t.sendFile` 在这个骨架里就是多读一个 `input.files`：把 base64 的文件按你应用接口的方式放进请求体即可；应用接口不收文件就忽略它。
