[查看结果](/zh/guides/viewing-results)讲「用」：官方两扇门 `niceeval show`（终端）和 `niceeval view`（网页）怎么看。本页讲「写」：官方摆法不够时，怎么写一份自己的报告——考试成绩单、代码行数榜、质量 × 成本 frontier。

一份报告就是一个报告文件。你不用打开结果目录、不用写渲染代码、不用起自己的应用：`niceeval show` / `niceeval view` 本体就是宿主——替你打开结果、把数据注入进来，你用官方组件和 `Row` / `Col` 摆版面，写完把文件路径递给 `--report`，终端和网页两扇门就都认它——官方的证据室深链、`--run` 历史切换、静态导出，自定义报告全部原样享有。

## 官方默认报告也是一份报告文件

`niceeval view` 和 `niceeval show` 不传 `--report` 时渲染的默认报告不是私有实现，而是内置的一份报告文件：`niceeval view` 就等于 `niceeval view --report <内置默认报告>`，`show` 同理。我们自己用的零件和给你的完全一样，没有私有数据通道，默认报告上的每个数字你都能用同样几行代码原样重算出来。

|  | 网页（人看） | 终端（agent 和你看） |
| --- | --- | --- |
| 官方摆法 | `niceeval view` | `niceeval show` |
| 自定义摆法 | `niceeval view --report reports/exam.tsx` | `niceeval show --report reports/exam.tsx` |

四个格子是同一个宿主。自定义只有三个层次，逐层深入：

1. **换摆法**：用官方组件和 `Row` / `Col` 重摆版面——大多数「要自己的口径」到这层就够了。
2. **换口径**：`defineMetric` 定义自己的指标，`flag()` 把 experiment 声明的变量当维度和轴。
3. **换形态**：官方组件表达不了的展示，用 `defineComponent` 写自己的组件——网页怎么渲染、终端字符怎么排，两个面你都说了算。

## 一份报告 = 一个报告文件

先交代唯一的前置：报告文件是 `.tsx`，写它的项目要装 `react`（写自定义组件的 web 面还要 `@types/react`），tsconfig 里 `compilerOptions.jsx` 设为 `"react-jsx"`。裸跑 `niceeval show` / `niceeval view` 不需要这些——只有自己写报告文件才需要。

报告基座是 `defineReport`：宿主打开结果目录（含 `--run` 指定历史 run）、按官方口径挑好快照，注入给你的函数；你只负责折数据和摆积木：

```tsx
// reports/exam.tsx —— 一份定义，两扇门共用
import {
  defineReport, Col, Section,
  DefaultReport, Scoreboard,
} from "niceeval/report";

export default defineReport(async ({ selection }) => (
  <Col>
    <DefaultReport />
    {/* 官方水位原样铺进来，不要就删这行 */}
    <Section title="考试成绩单">
      <Scoreboard data={await Scoreboard.data(selection, { rows: "agent", subjects: "evalGroup" })} />
    </Section>
  </Col>
));
```

```bash
niceeval show --report reports/exam.tsx    # 终端：同一棵树走文本面
niceeval view --report reports/exam.tsx    # 网页：同一棵树走网页面，attempt 深链直达证据室
```

注入的上下文只有两样，全部是[结果数据 API](/zh/guides/results-data) 的原语，宿主没有私有通道：`results` 就是 `openResults(".niceeval")` 的返回（实验、历次快照、attempt 级 `diff()`、`trace()` 都在上面）；`selection` 就是 `results.latest()` 的返回选集——宿主按「每个实验取最新」挑好的现刻水位，与默认报告同口径，快照在 `selection.snapshots`，挑选警告在 `selection.warnings`。组件的 `data` 函数直接吃选集，警告随行——`RunOverview` 会把残缺快照的警告如实显示，不用你另外接线；默认挑法不合口径时，拿 `results` 自己挑，手工挑的 `Snapshot[]` 数组同样能喂给每个 `data` 函数。

命令行的范围先作用在挑选上，报告拿到的就是范围内的水位：位置参数的 eval id 前缀收窄选集覆盖的 eval（覆盖警告的分母同样收窄到范围内），`--run` 把结果根换成指定目录，`--experiment` 让选集只留该实验。`--history` 与 `--report` 互斥——趋势在报告里用 `exp.snapshots` 自己摆；证据切面（`--transcript` / `--trace` / `--diff`）只看证据，不渲染报告。

`<DefaultReport />` 零 props、纯声明：官方水位只读瘦身条目（不碰 diff 这类重工件），宿主总是替你备好，摆不摆在你。它渲染的是**宿主注入的那份选集**的官方水位——跟随命令行范围，但不跟随你在报告里自己另挑的口径；这正是它的意义：当官方锚点，和你的口径并排对照。要「官方摆法 + 自选口径」，用它脚下的同一批组件自己摆——它没有任何私有零件。

页面里的每个组件都是**双面**的：网页面是 React 渲染，终端面是字符渲染，两面吃同一份算好的数据。你在报告文件里只见到组件本身，从不碰某一面——走哪扇门由宿主决定。官方组件覆盖常见摆法——榜单 `MetricTable`、逐题格子 `MetricMatrix`、分组条形 `MetricBars`、成绩单 `Scoreboard`、质量 × 成本 frontier `MetricScatter`、趋势线 `MetricLine`、成对对比 `DeltaTable`、失败清单 `CaseList`——每个组件回答什么问题、怎么调用、终端字符输出长什么样，逐个见[报告组件](/zh/guides/report-components)。它们在宿主里自动接上证据室：网页面的格子深链到 attempt 弹窗，终端面在条目下印出 `niceeval show <eval id>` 下钻命令。

默认报告没有特权：`<DefaultReport />` 只是不传 `--report` 时渲染的那份内置报告，本身也是一个普通组件。「官方水位 + 自己的口径」像上面那样把它摆进自己的树；不要官方水位就不摆——组合发生在你的 tsx 里，没有 tab、没有注册表、没有配置。

## 排版：Row 与 Col 在两个面都成立

排版原语也是双面组件，一次摆放，两个面各自成立：

- **`<Col>`**：纵向依次排列——网页是块级堆叠，终端是逐块输出。
- **`<Row>`**：并排——网页是横向排布，终端是字符分栏；终端宽度不够时自动降级为纵向，不硬挤。
- **`<Section title="…">`**：带标题的块，网页是标题层级，终端是标题行加缩进。
- **`<Text>`**：说明文字，网页是段落，终端是折行文本。

```tsx
<Row>
  <Scoreboard data={board} />
  <MetricTable data={bench} />
</Row>
```

```text
$ niceeval show --report reports/exam.tsx
考试成绩单                          │  benchmark 榜
agent  总分       algebra  geometry │  agent  pass  cost
bub    86.5/100   45/50    41.5/50  │  bub    87%   $0.42
codex  78.0/100   40/50    38/50    │  codex  80%   $0.51
```

页面树里只放双面组件和排版原语，不放裸 HTML 标签——终端面没法渲染一个 `<div>`。要自由内容，说明文字用 `<Text>`，更自由的走下面的自定义组件。

## 换口径：自定义指标

默认报告里那张逐实验指标表的口径拆开就是 `MetricTable.data(selection, { rows: "experiment", columns: [passRate, costUSD, durationMs] })`——每个组件的数据算法挂在它自己身上，组件名后打个点就找到；换口径只是把 columns 换成自己的指标：

```tsx
// reports/golf.tsx —— code-golf：谁写出能用的代码，谁写得短
import {
  defineReport, Col, MetricTable,
  defineMetric, passRate, costUSD,
} from "niceeval/report";

// 项目自己的口径：只比能用的代码的行数
const codeLines = defineMetric({
  name: "code-lines",
  label: "代码行数",
  better: "lower",
  unit: "lines",
  where: (a) => a.result.outcome === "passed",
  async value(attempt) {
    const diff = await attempt.diff();
    if (!diff) return null;
    return Object.values(diff.generatedFiles)
      .reduce((n, src) => n + src.split("\n").length, 0);
  },
});

export default defineReport(async ({ selection }) => (
  <Col>
    <MetricTable data={await MetricTable.data(selection, {
      rows: "agent",
      columns: [passRate, codeLines, costUSD],
      sort: passRate,
    })} />
  </Col>
));
```

```text
$ niceeval show --report reports/golf.tsx
agent   pass rate   code lines   cost
bub     87%         312 lines    $0.42
codex   80%         355 lines    $0.51
```

格子里的终值是**两级折叠**出来的：同一道题的多个 attempt 先折成题级值，再跨题折成格子值，两级默认都是平均。分两级不是形式主义——失败的题天然比通过的题 attempt 多（重试跑满、通过即停），平铺求均值会让分数和重试策略纠缠在一起。两级都能在 `defineMetric` 的 `aggregate` 里换：`aggregate: { perEval: "max", across: "mean" }` 就是 pass@k（题内取最好一次，跨题取占比）。

两个面继承同一套诚实契约：排序方向随指标的 `better`，覆盖不全的格子带 `12/15` 角标（15 个 attempt 里 12 个测得了该指标），缺数据渲染成 `—` 而不是 0。指标只定义一次，两个面共用——人在网页上看到的数字，就是 agent 在 stdout 里读到的数字，判断口径永远一致。给 agent 的指引也只多一行：跑 `niceeval show --report reports/golf.tsx`，读 stdout。

## 变量来自配置，不来自命名

画「并行 agent 数 × 模拟延迟 × 得分」这类 scaling 趋势时，图上的变量不该编码进 experiment id（`ultra-16agents-300ms`），再在报告里解析字符串抠出来——那是约定不是配置，改个命名整张图就散。变量的家是 experiment 文件的 `flags`：一文件一个档位，把变量声明成数据：

```ts
// experiments/ultra/agents-16.ts
import { defineExperiment } from "niceeval";
import { bub } from "../../agents/bub.ts";

export default defineExperiment({
  agent: bub({ mode: "ultra" }),
  flags: { agents: 16, latencyMs: 300 },
});
```

报告侧用 `flag()` 把声明的值直接当维度或轴用，不解析任何名字：

```tsx
<MetricLine data={await MetricLine.data(selection, {
  x: flag("latencyMs", { label: "Simulated latency", unit: "ms" }),
  series: flag("agents", { label: (v) => `${v} agents` }),
  y: passRate,
})} />
```

`flag()` 用在维度槽（`series` / `rows` / `columns` / `points`）就按声明值分组，用在轴槽（`x`）就要求数值并驱动刻度；内置维度（`agent`、`model`、`experiment`、`evalGroup`……）覆盖不了的变量都走它。没有声明该 flag 的 experiment 不猜：分组时如实归入「未配置」一组，作轴时该点不画、注脚报数。flags 随快照落盘，所以历史 run 也能按当时声明的变量重摆——报告读的是配置的事实，不是命名的巧合。

## 换形态：自定义组件，一个定义两个面

官方组件表达不了的展示——通过率条形、预算燃尽、项目自己的徽章——用 `defineComponent` 声明两个渲染面，和 `defineExperiment` / `defineMetric` / `defineReport` 同一个家族：

```tsx
// reports/passbars.tsx
import { defineReport, defineComponent, Col, Style, MetricTable, passRate } from "niceeval/report";

interface BarRow { key: string; ratio: number | null; display: string }

const PassBars = defineComponent<{ rows: BarRow[] }>({
  web({ rows }) {
    return (
      <ul className="passbars">
        {rows.map((r) => (
          <li key={r.key}>
            <span>{r.key}</span>
            <i style={{ width: `${(r.ratio ?? 0) * 100}%` }} />
            <b>{r.ratio === null ? "—" : r.display}</b>
          </li>
        ))}
      </ul>
    );
  },
  text({ rows }, { width }) {
    const bar = (n: number) => "█".repeat(Math.round(n * 10)).padEnd(10, "░");
    return rows
      .map((r) => `${r.key.padEnd(8)} ${r.ratio === null ? "—".padEnd(10) : bar(r.ratio)} ${r.display}`)
      .join("\n");
  },
});

export default defineReport(async ({ selection }) => {
  const board = await MetricTable.data(selection, { rows: "agent", columns: [passRate] });
  const rows = board.rows.map((r) => ({
    key: r.key,
    ratio: r.cells[passRate.name].value,   // 格子键锚在指标对象上,不裸写字符串
    display: r.cells[passRate.name].display,
  }));
  return (
    <Col>
      <Style>{`.passbars li { display: flex; gap: 8px; } .passbars i { background: #4a7; height: 12px; }`}</Style>
      <PassBars rows={rows} />
    </Col>
  );
});
```

```text
$ niceeval show --report reports/passbars.tsx
bub      █████████░ 87%
codex    ████████░░ 80%
claude   —          —
```

组件的契约：

- **计算在报告函数体里，渲染面是纯函数。** 读 attempt 句柄、`await` 折数据都发生在 `defineReport` 的函数体里；`web` 和 `text` 只认算好的数据，零 IO、同步——这条边界让同一棵树能被烘进静态导出。
- **面收第二参上下文。** `text(props, ctx)` 的 `ctx.width` 是可用列宽（`Row` 分栏后会变窄），`ctx.attemptCommand(ref)` 生成下钻命令；`web(props, ctx)` 的 `ctx.attemptHref(ref)` 生成证据室深链——自定义组件和官方组件通到同一间证据室。
- **网页面静态渲染，不 hydrate。** 宿主在计算侧把 `web` 面渲染成静态 HTML，不打包你的代码进查看器：交互用普通链接和 `<details>`，与官方组件同一条静态契约。
- **样式随树带走。** 静态导出不打包你的代码，`className` 引用的 CSS 用内置原语 `<Style>{css}</Style>` 放进页面树——web 面吐 `<style>` 标签，text 面渲染为空，上面的例子就是这么给 `.passbars` 上样式的。
- **诚实契约同样适用。** 缺数据渲染 `—` 不补 0，截断如实标注剩余数量——上面例子里 `claude` 没有样本就是 `—`。

## 发布：导出即静态站

发布自定义报告和发布官方查看器是同一个动作——`--out` 加上 `--report`：

```bash
niceeval view --report reports/exam.tsx --out site
```

产物是纯静态文件：你的报告页是首页，证据室（transcript、trace、代码视图）在同一站内，报告里的每个数字点进去就是证据。组件不 hydrate，产物不需要任何客户端 JS，交给任何静态托管即可。CI 上没有 `.niceeval/` 时，先用 `copySnapshots` 把快照瘦身进仓库再导出，流程见[查看结果](/zh/guides/viewing-results#导出与静态托管)。

双面组件的网页面就是普通 React 组件，`data` 函数就是普通 TS 函数——想把某一块指标表嵌进已有的内部面板，import 组件喂数据就行。那是零件的复用，不是另一套报告系统：报告的家在官方宿主。算数据与渲染分离部署时（CI 落 JSON、另一个应用 fetch），两侧锁同一个 niceeval 版本是硬要求——组件数据不带版本戳，兼容性跟随包版本。

## 界线：内置命令不长配置

报告槽只有一个，`--report` 收显式文件路径——没有 `reports/` 目录自动发现、没有插件注册表、没有配置文件。自定义指标和自定义组件都住在你的报告文件里，随文件一起递入，宿主不为它们长任何注册面。默认报告只是报告槽的出厂填充，你的报告和它是同级实现。
