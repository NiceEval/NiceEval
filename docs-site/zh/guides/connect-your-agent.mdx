把一个被测对象接进 [NiceEval](https://niceeval.com/)，就是给它写一个 **adapter**：`defineAgent` 包住一个 `send` 函数——接一次输入，驱动你的 agent，返回这一轮结果。接入的全部工作就这一个文件。

本文只回答两个问题：**这个文件和它的两样配套（experiment、eval）怎么跑通**，以及**参数怎么从 experiment 流到 Adapter、再流到你的应用**。事件流、多轮、HITL、tracing 都是之后的可选增量，各有专章，文末给了一张地图。

## 跑通最小接入

假设项目已经 `npx niceeval init` 过（有 `niceeval.config.ts` 和 `evals/` 目录）。三个文件各管一件事：**adapter 连被测系统，experiment 定评谁怎么跑，eval 写交互和断言。**

**1. 写 adapter。** 最小接入只填 `status` 和 `events`：你的 agent 说了什么，放一条 `message` 事件。

```ts
// agents/my-bot.ts
import { defineAgent } from "niceeval/adapter";

export default defineAgent({
  name: "my-bot",
  async send(input, ctx) {
    const r = await fetch("http://localhost:3000/chat", {
      method: "POST",
      headers: { "content-type": "application/json" },
      body: JSON.stringify({ message: input.text, model: ctx.model }),  // ← experiment.model 经 ctx.model 到这里
      signal: ctx.signal,
    });
    const body = await r.json();
    return {
      status: r.ok ? "completed" : "failed",
      events: [{ type: "message", role: "assistant", text: body.reply }],
    };
  },
});
```

URL 先写死没关系——怎么把它从 experiment 传进来，见下文[「参数怎么进」](#参数怎么进：experiment-声明，adapter-消费)。`send` 的完整契约（`TurnInput` / `AgentContext` / `Turn` 各字段）见 [Adapter](/zh/concepts/adapter)。

就算 agent runtime 和 eval 在同一个代码库里，也照样走 HTTP，不要把 `fetch` 换成进程内的函数直调：

- **直调评的不是用户走的那条链路。** HTTP 层、序列化、中间件、流式传输全被绕过，eval 通过不代表线上行为正确。
- **进程不隔离，结果不可复现。** 被测代码的崩溃、全局状态、内存泄漏会连累运行器；并发跑多条 eval 时共享状态互相污染。
- **超时与取消会失效。** `ctx.signal` 挂在请求上才能真正掐掉一轮跑飞的 agent；进程内的函数调用停不下来。
- **adapter 会被焊死在这个代码库里。** 走 HTTP 的 adapter 换个 `baseUrl` 就能对着本地 / 预发 / 生产跑（见下文两个 experiment 文件的写法）；直调做不到。

起一个本地 dev server 的成本只是一条命令，上面这些换不来。

**2. 注册进一个 experiment：**

```ts
// experiments/my-bot.ts
import { defineExperiment } from "niceeval";
import myBot from "../agents/my-bot.ts";

export default defineExperiment({
  description: "my-bot 基线",
  agent: myBot,
  model: "gpt-4o",
  runs: 1,
});
```

**3. 写第一条 eval，然后跑：**

```ts
// evals/refund-policy.eval.ts
import { defineEval } from "niceeval";
import { includes } from "niceeval/expect";

export default defineEval({
  description: "退款政策问答",
  async test(t) {
    await t.send("你们的退款政策是什么?");
    t.succeeded();
    t.check(t.reply, includes("30 天"));
    t.judge.autoevals.closedQA("回答是否说明了退款期限?").atLeast(0.7);
  },
});
```

```bash
npx niceeval exp my-bot        # 跑这个 experiment 下的全部 eval
npx niceeval exp my-bot refund # 只跑 ID 以 refund 开头的
npx niceeval view              # 本地查看器里看结果
```

**跑通的样子**：终端里每条 eval 一行判决，末尾汇总 `N passed, N failed`；`npx niceeval view` 里能看到每条 eval 逐轮的输入、事件和评分明细。

没跑通时，按报错的位置分三类排查：

- **`fetch` 直接抛错**（连接被拒等）：应用没起来，或 `send` 里的 URL 不对——先用 `curl` 对那个接口发一次同样的请求确认。
- **`t.succeeded()` 没过、本轮判决是 failed**：请求发出去了但应用回了非 2xx。在 `send` 里用 `ctx.log()` 打印响应体，`niceeval view` 的本轮日志里能看到。
- **只有内容断言没过**：接入本身已经通了——在 `view` 里对照 `t.reply` 的实际值，调断言或调应用。

到这里接入已经完成：文本断言和 judge 评分都能用了。想要更多断言（工具、多轮、审批流），见文末的增量地图。

## 参数怎么进：experiment 声明，Adapter 消费

配置归属只有两条通道，分清它们，接入就不会乱：

1. **静态配置走 Adapter 工厂。** URL、鉴权、协议细节这类"这个环境长什么样"的配置，作为工厂参数写在 experiment 文件里——`defineExperiment` 的 `agent` 字段放的是**已经配置好的实例**。
2. **每轮动态值走 `ctx`。** experiment 声明的 `model`、`flags`，运行器每轮经 `ctx` 原样递给 `send`；Adapter 不解释它们的含义，只随请求转发给应用。

两条通道拼在一起，就是把第一步的 my-bot 从「写死 URL 的实例」改成「收配置的工厂」：default export 换成一个返回 `defineAgent(...)` 的函数，`send` 里写死的地方换成读工厂参数；experiment 声明的 `model`、`flags` 每轮从 `ctx` 到达，`send` 里随请求转发：

```ts
// agents/my-bot.ts —— 工厂：静态配置进闭包
import { defineAgent } from "niceeval/adapter";
import type { Agent } from "niceeval/adapter";

export function myBot(options: { baseUrl: string }): Agent {
  return defineAgent({
    name: "my-bot",
    async send(input, ctx) {
      const r = await fetch(`${options.baseUrl}/chat`, {   // ← 工厂配置：连哪
        method: "POST",
        headers: { "content-type": "application/json" },
        body: JSON.stringify({
          message: input.text,
          model: ctx.model,     // ← experiment.model：应用接口收模型选择就转发
          flags: ctx.flags,     // ← experiment.flags：原样交给应用，应用按 flag 切换
        }),
        signal: ctx.signal,     // ← 运行器的超时与取消：挂到每个请求上
      });
      const body = await r.json();
      return {
        status: r.ok ? "completed" : "failed",
        events: [{ type: "message", role: "assistant", text: body.reply }],
      };
    },
  });
}
```

鉴权 header、协议开关这类同属静态配置，一样加进 `options`。experiment 侧对应两处小改：具名导入工厂，`agent` 字段从「引用实例」变成「调用工厂」：

```ts
// experiments/my-bot.ts —— 参数的声明处
import { defineExperiment } from "niceeval";
import { myBot } from "../agents/my-bot.ts";

export default defineExperiment({
  description: "my-bot 基线",
  agent: myBot({ baseUrl: "http://localhost:3000" }),  // ← 静态配置：实例化时传
  model: "gpt-5.4",                                    // ← 动态值：经 ctx.model 到 send
  flags: { promptVariant: "concise" },                 // ← 动态值：经 ctx.flags 到 send
});
```

`ctx` 上每轮可能用到的字段和消费方式：

| `ctx` 字段 | 谁给的 | Adapter 怎么消费 |
|---|---|---|
| `signal` | 运行器（超时与取消） | 挂到发出的每个请求上 |
| `model` | experiment 的 `model` | 应用接口收模型选择就随请求转发；不收就忽略 |
| `flags` | experiment 的 `flags` | 原样转发（请求体、header 都行），应用按 flag 切换变体 |
| `telemetry` | 配置了 OTel 接入时出现 | 只碰 `headers`：每轮新的 W3C `traceparent`，spread 进请求头。接收端点每次运行都一样，在 `defineConfig` 里固定、应用启动时指向它，不从 send 传——见 [OTel 接入](/zh/guides/connect-otel) |
| `session` | 运行器（每条会话线一份） | 会话续接与 HITL 停轮现场的存取器都在它上面：`history()`、`id` / `capture()`、`hold()` / `take()`，见[写 send](/zh/guides/write-send) |
| `log(msg)` | 运行器 | 写进本轮日志，`niceeval view` 里可见 |

同一个 agent 对着本地和生产分别跑，就是两个 experiment 文件、两份工厂参数：

```ts
// experiments/local.ts
export default defineExperiment({
  agent: myBot({ baseUrl: "http://localhost:3000" }),
  model: "gpt-5.4",
});

// experiments/prod.ts
export default defineExperiment({
  agent: myBot({ baseUrl: "https://api.example.com" }),
  model: "gpt-5.4",
});
```

```bash
npx niceeval exp local
npx niceeval exp prod
```

不要把 URL 放进 CLI 位置参数——experiment 名之后的位置参数只用于过滤 eval ID。experiment 的完整字段（`runs`、`budget`、并发、`sandbox`）见[写实验](/zh/guides/write-experiment)。

## 之后的增量：一张地图

最小接入之后，每给 adapter 加一个增量就解锁一组断言，已写的 eval 不用改：

| 想解锁什么 | 给 adapter 加什么 | 去哪看 |
|---|---|---|
| 工具断言（`calledTool` / `toolOrder` / 负断言） | 把应用返回映射成标准事件流 | [写 send](/zh/guides/write-send)、[事件流参考](/zh/reference/events) |
| 多轮对话、`t.newSession()` 隔离 | 接上 `ctx.session`：`history()` 或 `id` + `capture()` | [写 send](/zh/guides/write-send) |
| 审批流（HITL，人工介入） | 停轮返回 `waiting` + `input.requested`，回答轮续跑 | [HITL](/zh/concepts/hitl) |
| `niceeval view` 的调用瀑布图 | 应用把 OTel span 发给 NiceEval（不影响断言） | [OTel 接入](/zh/guides/connect-otel) |
| feature A/B 对比 | 应用把变体暴露成 `flags` 可切换的配置 | [Tier](/zh/concepts/tier)、[写实验](/zh/guides/write-experiment) |

这些增量落在哪一档、各档买到什么，见 [Tier](/zh/concepts/tier)。
