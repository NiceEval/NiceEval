A coding agent can write NiceEval configuration and evals from a spec, and can also keep improving the system under test based on run results. The whole process only needs repo files and bash — no browser, and no extra skill to install.

## Point the agent at the installed docs first

NiceEval ships its Chinese docs under the npm package's `docs-site/zh/` directory, and publishes an `INDEX.md` at the package root for a coding agent to use. The agent should read `node_modules/niceeval/INDEX.md` first, then follow the index into whichever page the current task needs — not rely on training data or some other version of the examples found online. That keeps the API and CLI pinned to the installed version.

`npx niceeval init` scaffolds the config and writes a managed block of guidance into the project's `AGENTS.md`. If the project only has a `CLAUDE.md`, it writes there instead; when neither file exists, it creates a new `AGENTS.md`. Running `init` again after upgrading NiceEval refreshes the managed block.

A starting task for the agent can be written directly as:

```text
Start by reading node_modules/niceeval/INDEX.md, then follow the index to read
whatever docs the current task needs, then write niceeval.config.ts, the
experiment, and the eval to match the spec.
Don't guess NiceEval's API from training data.
```

An agent typically picks these entry points by task:

| Task | Bundled doc |
| --- | --- |
| Initialize a project | `docs-site/zh/tutorials/quickstart.mdx` |
| Write an eval | `docs-site/zh/explanation/evals.mdx` |
| Define an experiment | `docs-site/zh/tutorials/write-experiment.mdx` |
| Connect the agent under test | `docs-site/zh/explanation/adapter.mdx` |
| Configure a Sandbox | `docs-site/zh/tutorials/sandbox-agent.mdx` |
| Interpret run results | `docs-site/zh/tutorials/viewing-results.mdx` |

## Closing one feedback loop with bash

The goal of the loop isn't to make a command exit green — it's to form and verify a hypothesis about where the failure comes from: the system under test, the eval itself, or the runtime environment. Iterate in the order below.

<Steps>
  <Step title="Run the experiment">
    ```bash
    npx niceeval exp local --output agent
    ```

    Read the exit code first: `0` means every eval passed; `1` means at least one eval failed or errored; `2` means NiceEval itself failed to complete the run. The exit code decides whether to keep going; the console text is for locating the cause.
  </Step>
  <Step title="Read the failures">
    ```bash
    npx niceeval show
    npx niceeval show @1k2m9qtr
    ```

    The first command shows each experiment's current pass rate, cost, and duration, plus a compact attempt locator for every eval — the locator itself is the entry point to the evidence, so the list does not encode which evidence is available. The second command opens the selected attempt directly; the `available:` block at the end of that page lists only the evidence commands this attempt actually has.
  </Step>
  <Step title="Read evidence for your question">
    ```bash
    npx niceeval show @1k2m9qtr --execution
    npx niceeval show @1c3h6tbn --diff
    ```

    `--execution` merges the agent's output with the trace: the standard event stream supplies messages, thinking, tool call/result, and skill loads; OTel, when it can be correlated, adds start time, duration, parent/child relationships, and error status onto those same nodes. Without OTel the steps are still complete — they just carry no timing.

    With no evidence flag, `show @<id>` is the failure diagnosis front page. It first lists each failed assertion's group, matcher, expected value, received value, reason, and source location, then a summary of execution, lifecycle-stage timing, and file changes. The agent should read this page first, and only open the matching evidence when it needs to answer "why was this value produced".

    ```text
    $ niceeval show @1k2m9qtr
    @1k2m9qtr · weather/brooklyn · compare/codex-gpt-5.4 · failed
    snapshot 2026-07-09T10-00-00-000Z-x1f2 · attempt 2 · 41.2s · 12.3k tokens · $0.04
    eval source: evals/weather/brooklyn.eval.ts · sha256:8d47…

     8  export default defineEval({
     9    description: "Brooklyn weather query",
    10    async test(t) {
    11      const turn = await t.send("What's the weather in Brooklyn today?");
    12 ✗    turn.calledTool("get_weather");
           calledTool("get_weather") · received 0 tool calls
    13 ✓    turn.succeeded();
    14 ✗    t.check(turn.message, judge("reply is based on real-time data"));
           judge("the answer is based on real-time data") · 0.2 / 1 · reply invents a temperature without any tool call
    15    },
    16  });

    assertions: 1 passed · 1 gate failed · 1 soft below target

    execution: 3 events · 0 skill loads · 0 tool calls · 1 AI messages
    timing: eval.run 40.4s · scoring.evaluate 0.5s · teardown +0.2s

    changes · diff unavailable
    reason: this attempt did not produce workspace file changes

    full eval source: …/weather/brooklyn/a2/sources.json
    available:
      niceeval show @1k2m9qtr --source
      niceeval show @1k2m9qtr --execution
      niceeval show @1k2m9qtr --timing
    ```

    `--execution` lays out AI messages, skill loads, tool calls, and tool results as one execution tree. It only shows events an agent can make sense of; SDK / runtime spans that correlate to none of those events are not printed line by line — only their omitted count is reported, and the `trace.json` path is kept. The attempt below has OTel, so the nodes that could be correlated carry both relative time and duration:

    ```text
    $ niceeval show @1k2m9qtr --execution
    @1k2m9qtr · weather/brooklyn · compare/codex-gpt-5.4 · failed · OTel timing

    USER
      What's the weather in Brooklyn today?

    ASSISTANT
      Let me check the weather tool's usage notes first.

    TOOL · get_weather  0.8s · 1.8s
      input
        {"location":"Brooklyn"}
      result · completed
        {"temperatureC":18,"condition":"rain"}

    ASSISTANT  2.8s · 1.2s
      Brooklyn is currently about 18°C with rain.

    total 4.0s · 1 skill load · 1 tool call · 2 AI messages
    full events: …/weather/brooklyn/a2/events.json
    69 unlinked telemetry spans omitted; inspect the OTel trace for framework timing.
    full OTel trace: …/weather/brooklyn/a2/trace.json
    ```

    Without OTel, the same steps still render from the standard event stream — only the time column and the waterfall relationship are gone; a missing timing column is never mistaken for "nothing executed":

    ```text
    $ niceeval show @1nx4dpwq --execution
    @1nx4dpwq · weather/brooklyn · compare/bub-gpt-5.4 · failed

    USER
      What's the weather in Brooklyn today?

    ASSISTANT
      Let me check the weather tool's usage notes first.

    TOOL · get_weather
      input
        {"location":"Brooklyn"}
      result · completed
        {"temperatureC":18,"condition":"rain"}

    ASSISTANT
      Brooklyn is currently about 18°C with rain.

    timing unavailable · OTel trace was not collected
    full events: …/weather/brooklyn/a2/events.json
    ```

    A diff is the file changes the tested agent made in the Sandbox workspace, not a diff of the eval source itself. Only Sandbox evals collect a diff — for a non-Sandbox eval, or when the agent genuinely touched no files, the attempt page's `available:` list omits `--diff`. It defaults to a file-level summary, so a large patch doesn't get dumped straight into the agent's context; `--diff=<file>` expands a single file, and the raw artifact path is always kept alongside it:

    ```text
    $ niceeval show @1c3h6tbn --diff
    @1c3h6tbn · fixtures/button · compare/bub-gpt-5.4 · errored

    M src/components/Button.tsx   +42 −3
    A src/components/Button.css   +18

    2 files changed · +60 −3
    full diff: …/fixtures/button/a1/diff.json
    next: niceeval show @1c3h6tbn --diff=src/components/Button.tsx
    ```

    Use this set of outputs to check whether you have enough information:

    | Question you're trying to answer | Entry point | Output must include |
    | --- | --- | --- |
    | Quickly tell what happened in one attempt | `niceeval show @<id>` | Eval assertions, execution steps, a lifecycle-stage timing summary, a diff summary, and which blocks are available |
    | What the eval actually checked, and why a given gate/soft passed or failed | `niceeval show @<id> --source` | The eval source at run time, its source hash, the line each assertion sits on, severity, score, and reason |
    | What the agent did, what it called, where the time went | `niceeval show @<id> --execution` | Messages, thinking, skill loads, tool calls and results; with OTel, timing, parent/child relationships, and error status on the same nodes |
    | How an attempt's time breaks down phase by phase | `niceeval show @<id> --timing` | A unified time tree from lifecycle → hook/turn → shell → OTel; an errored attempt marks the deepest known failing node |
    | What the Sandbox workspace's files turned into | `niceeval show @<id> --diff` | A file summary, added/removed line counts, the actual patch, and the original diff path; explicitly unavailable when there's no file workspace |
  </Step>
  <Step title="Form a hypothesis, then fix">
    Based on the evidence, change only the side that's most likely at fault:

    | What you see | Check first |
    | --- | --- |
    | The reply or tool call doesn't match the spec | The system under test, its prompt, its tool definitions |
    | The `diff` doesn't match what the task asked for | The tested agent's implementation strategy, or its runtime environment |
    | Correct behavior gets rejected by a gate | The eval's assertions, Fixtures, setup |
    | Timeout, auth, or Sandbox errors | Experiment configuration, the Adapter, the Sandbox |
    | The same eval passes sometimes and fails other times | `--history`, `runs`, and stability issues |

    Don't loosen a correct assertion just to turn it green. Write down which evidence supports which conclusion before touching the code.
  </Step>
  <Step title="Rerun locally, verify the hypothesis">
    ```bash
    npx niceeval exp local weather/brooklyn --output agent --force
    npx niceeval show weather/brooklyn
    ```

    The positional argument narrows the experiment by eval id prefix. While debugging the same failure, add `--force` to make sure the fix you just made actually triggers a new run instead of replaying a cached one. Then use `show` to verify the verdict, assertions, and evidence changed the way you expected.
  </Step>
  <Step title="Full rerun to confirm no regression">
    ```bash
    npx niceeval exp local --output agent --force
    npx niceeval show
    ```

    A local pass only confirms the current hypothesis. Before calling it done, force a full rerun — only when the command exits `0`, and `show` shows no new failures or errors, is the loop actually complete.
  </Step>
</Steps>

## What the agent should read from the output

`--output agent` only appends low-frequency checkpoints to stderr while the run is in flight (a liveness signal, not a results source) and prints one bounded handoff block to stdout at the end — that block is what the agent should parse:

```text
NICEEVAL RESULT failed
summary: 14 passed, 1 failed, 0 errored (0 reused)
snapshots:
  - .niceeval/local/2026-07-09T10-00-00-000Z-x1f2/
failures:
  - @1k2m9qtr weather/brooklyn [local]
    gate: calledTool("get_weather")
          received 0 tool calls
next:
  niceeval show @1k2m9qtr
  niceeval show @1k2m9qtr --execution
```

The agent should pick an attempt locator from `failures`, then run one of the `next` commands (`niceeval show @<id>` or the matching evidence flag) — not parse the low-frequency checkpoint lines on stderr while the run is in flight; those only signal that the process is still alive. When the failure count exceeds the cap (5 by default), the handoff expands only the first few and reports the total; the full list comes from the results snapshot. The results snapshot is the source of truth for programmatic reading:

```text
.niceeval/<experiment>/<snapshot>/
├── snapshot.json
└── <eval-id>/a<attempt>/
    ├── result.json
    ├── events.json
    ├── trace.json
    └── diff.json
```

- `snapshot.json` records the experiment's identity, run configuration, format version, and timestamp.
- `result.json` records that attempt's verdict, assertions, structured errors, diagnostics, and usage. Transient progress is never persisted.
- `events.json` holds the conversation and tool-call events, `trace.json` holds the call chain, and `diff.json` holds Sandbox file changes.
- When a kind of evidence doesn't exist, the corresponding file simply isn't generated. Trust what `show` reports as available rather than assuming every directory has every file.

`show`'s default view can be composed from multiple runs: it picks the latest verdict per experiment × eval, so after a local rerun you'll still see other evals' older results sitting alongside it. This view is for checking each eval's latest known verdict; confirming the overall result for one exact version of the code always requires a `--force` full run.

## When results get reused

Without `--force`, NiceEval compares the current fingerprint against the most recent result. The fingerprint is made of the eval's source and the run configuration — experiment id, agent, model, flags, Sandbox, timeout, strict, and similar settings.

| Most recent result vs. current input | This run's behavior |
| --- | --- |
| Same fingerprint, verdict `passed` or `failed` | Carries the old result forward, does not re-execute |
| Different fingerprint | Re-executes |
| Most recent result was `errored` or `skipped` | Re-executes |
| `--force` used | Ignores the old result, re-executes |

The system under test's source is not part of the fingerprint. After changing the implementation, even though behavior has changed, an old `passed` or `failed` can still get reused. So choose based on what you actually changed:

- Just want to look at existing results again: run `niceeval show` — no new cost.
- Changed the eval or the experiment configuration: rerun directly; the fingerprint change triggers the affected task on its own.
- Changed the system under test: use `--force` on the affected evals.
- Wrapping up this round of work: use `--force` on the whole experiment, to rule out regressions in other evals.

## Letting the agent converge on its own

This protocol can go straight into a task description. It works equally well for improving the system under test and for debugging the eval itself:

```text
Read node_modules/niceeval/INDEX.md, then follow the index to read whatever
docs relate to the task.
Run npx niceeval exp local --output agent, and decide the next step from the
exit code and the failure locators. For every failing eval, pick an attempt
locator from the report; then run niceeval show @<id> and choose --source,
--execution, --timing, or --diff depending on the question you're asking;
--timing expands the lifecycle into setup/teardown hooks, shell commands,
each turn's send, and correlated OTel model/tool spans, to answer where an
attempt's time went.
Write down a hypothesis for the failure, and decide whether to fix
the system under test, the eval, or the experiment environment. After fixing,
rerun the affected eval with --force and compare the new verdict and evidence.
If the same problem produces no new evidence or improvement for three rounds
in a row, stop and report back — don't reach for a green result by loosening
an assertion.
Once every local failure is cleared, do a full verification with
npx niceeval exp local --output agent --force; only exit code 0 counts as done.
```

Running a real agent can cost money. During experimentation, add `--budget <dollars>` to cap this round's cumulative spend; a budget only bounds a single command — it's not a substitute for the stop condition above.

A person and an agent can hand off the same round of work to each other at any point. The results an agent reads with `niceeval show`, a person can open in the browser with `npx niceeval view` — both read the same batch of artifacts. The full output formats, history selection, and web UI walkthrough are in [Viewing Results](/tutorials/viewing-results).
