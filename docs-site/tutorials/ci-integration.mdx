Evals should live in CI the same way tests do. They catch agent regressions on pull requests and track model, cost, and latency changes in nightly jobs.

## Exit codes

By default, [NiceEval](https://niceeval.com/) exits non-zero when any gate fails. In CI you usually want `--strict` so failure is explicit:

```bash
NICEEVAL_LANG=en npx niceeval exp ci \
  --output ci \
  --strict \
  --json .niceeval/ci-summary.json \
  --junit .niceeval/junit.xml
```

The `ci` profile never prints ANSI, a spinner, or a dynamic table. Its log is a single ordered stdout stream that only appends `start`, low-frequency heartbeats, failures/errors, diagnostics, and the final result — passed attempts are never printed one by one.

## GitHub Actions example

```yaml
name: evals
on:
  pull_request:
  workflow_dispatch:

jobs:
  evals:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: npm
      - run: npm ci
      - run: >-
          npx niceeval exp ci --output ci --strict
          --json .niceeval/ci-summary.json
          --junit .niceeval/junit.xml
        env:
          NICEEVAL_LANG: en
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

## JSON, JUnit, and structured errors

```text
niceeval: start total=24 configs=3 concurrency=10 reused=18
niceeval: errored locator=@12h8m4k1 eval=fixtures/button experiment=ci/codex phase=sandbox.create reason="E2B sandbox allocation failed after 5 attempts"
niceeval: result=failed passed=23 failed=0 errored=1 reused=18 duration=128s
niceeval: json=.niceeval/ci-summary.json
niceeval: junit=.niceeval/junit.xml
```

The exit code is the first-line red/green signal; JSON, JUnit, and the results snapshot are the complete machine interface — log lines are only for searching and annotation. An `errored` line carries a locator, the eval/experiment identity, the formal phase when known, and a one-line `reason` summary. The full cause, stack, and diagnostics live in the attempt's `result.json` — keep the artifacts and run `niceeval show @<locator>` to review them.

The JSON/JUnit files the CLI is explicitly told to write, and the default results artifact, are all required outputs: a write failure must turn the job red, not just log a warning and exit 0. To also report results to an experiment platform such as Braintrust, see [Reporters](./reporters).

## Discovery only

```bash
npx niceeval list
```

This is useful when you only want to confirm that eval files load and configuration is sane.

## Cache `.niceeval/`

You can cache `.niceeval/` in CI, but make sure the fingerprint covers every input that affects the result. For nightly benchmarks, keeping the full artifacts is usually more valuable.

## Control concurrency

```bash
npx niceeval exp ci --output ci --max-concurrency 2
```

Sandbox evals should stay conservative on a standard GitHub-hosted runner. Remote HTTP evals can usually run with higher concurrency if the service allows it.
