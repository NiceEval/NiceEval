Every [NiceEval](https://niceeval.com/) agent is an adapter: code that knows how to drive a specific backend and translate its output into the standard event stream. The runner only calls `agent.send(input, ctx)`.

![Full round trip of one t.send: the eval calls t.send, the runner assembles TurnInput and ctx, the adapter returns a Turn with standard events.](/images/agent-turn-roundtrip-en.svg)

## `defineAgent`

Use this for direct agent integrations:

```ts
import { defineAgent } from "niceeval/adapter";
```

`Agent.kind` produced by `defineAgent` is always `"remote"` (an internal discriminant field you never declare). There are no capability bits on `t` that need declaring to unlock ŌĆö filesystem assertions like `t.sandbox` exist only on agents constructed with `defineSandboxAgent` (`kind: "sandbox"`); everything else is judged purely by which events `send` actually returned and whether `ctx.session` was used. See the [capabilities reference](/reference/capabilities). The full field set is in "Agent and Adapter Context fields" below.

### Direct agent example

```ts
export default defineAgent({
  name: "echo",
  async send(input) {
    return {
      status: "completed",
      events: [{ type: "message", role: "assistant", text: input.text }],
    };
  },
});
```

## `input` (`TurnInput`)

<ResponseField name="text" type="string">
  Text passed by the current `t.send(...)`.
</ResponseField>

<ResponseField name="files" type="readonly InputFile[] | undefined">
  Files attached to this turn (images, etc.). Adapters that do not support multimodal input can ignore it.
</ResponseField>

<ResponseField name="responses" type="readonly InputResponse[] | undefined">
  Present only on answer turns (`t.respond` / `t.respondAll`): per-request structured answers, matched by `requestId`.
</ResponseField>

## `defineSandboxAgent`

Use this for coding agent CLIs. The produced `Agent.kind` is always `"sandbox"` ŌĆö filesystem assertions like `t.sandbox` and `t.sandbox.fileChanged()` unlock only on this kind of agent. The full field set is likewise in "Agent and Adapter Context fields" below, under `SandboxAgentDef`; the complete methods of `ctx.sandbox` (the `Sandbox` interface) are in the "Sandbox interface" section further below.

```ts
import { defineSandboxAgent } from "niceeval/adapter";

export default defineSandboxAgent({
  name: "my-cli-agent",
  async send(input, ctx) {
    await ctx.sandbox.runCommand("my-agent", ["run", input.text]);
    return { status: "completed", events: [] };
  },
});
```

## Registration

```ts
import { defineExperiment } from "niceeval";
import echo from "./agents/echo";

export default defineExperiment({
  agent: echo,
  runs: 1,
});
```

### `RemoteAgentDef`

#### `name`

```ts
name: string;
```

The agent's display name/identifier, passed as-is into `Agent.name` ŌĆö not a registry lookup key, only used for display, result attribution, and dedup fingerprinting.

#### `coverage`

```ts
coverage?: EvidenceCoverage;
```

This adapter's steady-state evidence coverage declaration (use the `completeCoverage` constant for full collection); omit it for all-channel unknown.

#### `setup`

```ts
setup?: AgentSetup;
```

Runs once per attempt (a remote agent has no real sandbox, so the runner passes in a stub `Sandbox` containing only metadata
like `workdir`/`sandboxId`/`otlpHost`/`stop` ŌĆö calling any other method throws. Do not call
file/command sandbox methods here). Commonly used for one-time setup like establishing a connection or auth;
does not return a value.

#### `tracing`

```ts
tracing?: AgentTracing;
```

OTLP export configuration: how the remote system under test sends traces to the endpoint (env-based injection / file-based configuration).

#### `spanMapper`

```ts
spanMapper?: SpanMapper;
```

Thin mapper from native span ŌåÆ canonical; omit it to fall back to the generic heuristic. Only affects the waterfall.

#### `send`

```ts
send(input: TurnInput, ctx: AgentContext): Promise<Turn>;
```

Once per turn: send one turn's prompt to the remote system under test (HTTP/SDK, etc.), and parse the response into events.

#### `teardown`

```ts
teardown?: AgentTeardown;
```

Cleanup before the run ends, run if and only if this attempt reached the `setup` point in time
(a `setup` throw does not exempt it); runs once in `finally`.

Once per turn: run the prompt (fresh / resume) and parse it into events.

#### `teardown`

```ts
teardown?: AgentTeardown;
```

Cleanup before the sandbox is destroyed, run if and only if this attempt reached the `setup` point in time

### `AgentContext`

#### `signal`

```ts
readonly signal: AbortSignal;
```

Soft-cancellation signal: merges the attempt timeout, run-level interruption (user Ctrl+C), and the eval's own interruption
request (see src/runner/attempt.ts). Adapters can optionally check it (or pass it straight to `fetch`) to exit early and
gracefully, but this is not the only hard boundary ŌĆö even if the adapter ignores it entirely, the runner still
force-finishes with `Effect.timeoutTo` as a fallback (stopping the sandbox container).

#### `model`

```ts
readonly model?: string;
```

The model name used for this attempt, passed through from the experiment's `model` field. Sandbox-type agents typically use it in setup to write config; remote-type agents typically use it in send to pick the model.

#### `reasoningEffort`

```ts
readonly reasoningEffort?: string;
```

Model reasoning effort; belongs alongside model ŌĆö decided by the experiment, and when omitted does not override the agent's native default.

#### `flags`

```ts
readonly flags: Readonly<Record<string, unknown>>;
```

The experiment's `flags` field, passed through as-is; its content and shape are entirely defined by the experiment author
(e.g. `{ webResearch: true }`, `{ systemPrompt: "..." }`). Adapters read fields from it by their own convention;
the framework itself does not interpret or validate its content. The name is deliberately distinct from the CLI-parsed
`flag` (run-level things like --timeout/--budget) ŌĆö the two are unrelated concepts.

#### `experimentId`

```ts
readonly experimentId?: string;
```

The experiment id derived from the path (same source as the result attribution `runWho` / `AgentRun.experimentId`);
undefined when the run does not go through an experiment (e.g. outside the CLI, constructing an `AgentRun` directly).
Typical uses: a `SandboxSpec.setup` hook isolating cross-attempt state per experiment (cache directory names,
snapshot tags, etc. partitioned by `ctx.experimentId`), or an adapter switching auth / routing per experiment. It is a
different dimension from `flags` (the concrete values of the experiment's conditions) ŌĆö this is only the stable identifier
of "which experiment is running", and carries no condition content.

#### `sandbox`

```ts
readonly sandbox: Sandbox;
```

Present on every agent: for sandbox-type agents it is the real sandbox handle the runner prepared from the
project/experiment configuration; for remote-type agents it is the stub produced by `createRemoteSandbox()` (only
metadata like `workdir`/`sandboxId`/`otlpHost`/`stop` ŌĆö calling any other method throws).

#### `session`

```ts
readonly session: AgentSession;
```

#### `telemetry`

```ts
readonly telemetry?: Telemetry;
```

Present only when OTel integration is configured (the agent's `tracing` block / the config's telemetry exists):
this run's OTLP trace receiving information (endpoint + env-based export env).
How to hand it to the CLI is declared by the agent's `tracing` block: env-based agents spread ctx.telemetry.env
into send; file-based agents write config in tracing.configure. For remote HTTP integrations, send
only needs to spread the headers into the request headers (a fresh traceparent per turn); the endpoint is fixed at
startup-time configuration (defineConfig(&#123; telemetry: &#123; port &#125; &#125;)) and is not passed here.

#### `progress`

```ts
progress(update: ProgressUpdate): void;
```

Scoped feedback: report what's happening right now (turn / tool / install progress). Short-lived state ŌĆö the
`human` profile updates the active row; `agent`/`ci` do not print it line by line, and it never lands in the
final result. Do not call it for every token/delta. The runner attributes it to whichever lifecycle phase the
callback is currently in (agent.setup / agent.run / agent.teardown); the caller cannot pretend to be in a
different phase (see docs/feature/experiments/library.md).

#### `diagnostic`

```ts
diagnostic(input: DiagnosticInput): void;
```

Scoped feedback: report a problem that should still be visible after the run ends (protocol downgrade,
incomplete data, cleanup issues). A permanent event ŌĆö it lands in the attempt's diagnostics and each profile's
permanent output; deduplicated by `dedupeKey`. Even a level of "error" does not change `Turn.status` /
verdict ŌĆö throw an exception when you cannot continue.

#### `log`

```ts
log(msg: string): void;
```

An alias for `progress({ message: msg })`, not a second channel (see "Attempt phases" in
`docs/feature/experiments/cli.md`). On a timeout failure, the most recent lines are merged into the result's
error information, to help locate where it got stuck.

`ctx.session` (`AgentSession`) is the state slot for one conversation line: every `send` on the same conversation line gets the same `ctx.session`, and a new conversation line (the eval's first turn / after `t.newSession()`) gets a fresh one. Accessors:

- `id?: string` / `capture(id): void` ŌĆö session continuation when the server keeps history: `id` is the session id recorded on this line (`undefined` on a new line), `capture` records the returned id (it only lands when none has been recorded yet).
- `history<TMsg>(): { get(): TMsg[]; commit(messages: TMsg[]): void }` ŌĆö session continuation when the client carries the full history.
- `hold<T>(state: T): void` / `take<T>(): T | undefined` ŌĆö storage for the HITL paused-turn scene; `take` clears on read (consume once).
- `state: Record<string, unknown>` ŌĆö escape hatch, starts as `{}`, never written by the framework.

Full contract in the [Adapter concept](/explanation/adapter#context-agentcontext).

## Sandbox interface

`ctx.sandbox` (`Sandbox`) on sandbox-type agents is the handle to the current isolated environment; `CommandOptions` are the options for `runCommand` / `runShell`:

### `Sandbox`

#### `workdir`

```ts
readonly workdir: string;
```

Absolute path to the project/workspace root inside the sandbox (the default cwd for agent commands, and where the git baseline is committed). Relative paths in every method resolve against this, and it's also where things land when `cwd`/`targetDir` is omitted.

#### `runCommand`

```ts
runCommand(cmd: string, args?: string[], opts?: CommandOptions): Promise<CommandResult>;
```

Run a single command; `args` is passed as a separate argv array and is not interpreted by a shell (no `&&`, pipes, or glob expansion).
Prefer this when you just want to run one executable, arguments come from external input, and you're worried about injection.

#### `runShell`

```ts
runShell(script: string, opts?: CommandOptions): Promise<CommandResult>;
```

Run a whole script, interpreted by a shell (bash), supporting `&&`, pipes, `$()`, redirection, etc.
Use this when you need to chain multiple commands or do conditional logic.

#### `readFile`

```ts
readFile(path: string): Promise<string>;
```

Read a file's text content inside the sandbox (UTF-8). Throws if the file doesn't exist rather than returning an empty string ŌĆö add your own `.catch()` if you need tolerance.

#### `fileExists`

```ts
fileExists(path: string): Promise<boolean>;
```

Check whether a path exists inside the sandbox. Semantics are not fully consistent across providers: only guaranteed reliable for regular files ŌĆö behavior for directory paths is not guaranteed consistent across providers.

#### `writeFiles`

```ts
writeFiles(files: Record<string, string>, targetDir?: string): Promise<void>;
```

Write several text files (content already in memory as strings); a text-specialized form of `uploadFiles`, landing in workdir when `targetDir` is omitted.

#### `uploadFiles`

```ts
uploadFiles(files: SandboxFile[], targetDir?: string): Promise<void>;
```

Batch-write several files, whose content can be text or a binary Buffer; lands in workdir when `targetDir` is omitted.

#### `uploadDirectory`

```ts
uploadDirectory(localDir: string, targetDir?: string, opts?: { ignore?: string[] }): Promise<void>;
```

Upload an entire local directory into the sandbox (reads local files recursively, then writes them via `uploadFiles`); `opts.ignore` is the exclusion rule, landing in workdir when `targetDir` is omitted.

#### `stop`

```ts
stop(): Promise<void>;
```

Destroy the compute resource the sandbox occupies (container/microVM). The sandbox is unusable after this; whether it's safe to call repeatedly varies by provider ŌĆö do not rely on it.

#### `sandboxId`

```ts
readonly sandboxId: string;
```

Stable identifier for this sandbox (each provider's native ID, e.g. a Docker container ID prefix); used to correlate session state for the same sandbox across calls, and for log display.

#### `otlpHost`

```ts
readonly otlpHost: string | null;
```

Target host for the local OTLP receiver.
- `string`: inside the sandbox, this hostname connects back to the host's OTLP port (e.g. Docker's `host.docker.internal`).
- `null`: the sandbox runs on remote cloud infrastructure (e.g. e2b/vercel) and cannot reach a local host port ŌåÆ tracing is skipped.
  Can be force-overridden with the `NICEEVAL_OTLP_HOST` environment variable (e.g. when configuring a tunnel).

#### `appendLog`

```ts
appendLog?(line: string): Promise<void>;
```

Optional: write a line into the container's "main log" (the one PID1 tails) ŌĆö so `docker logs` /
the Docker UI's Logs tab shows the agent's turn-by-turn activity live. Implemented by the docker provider; others can omit it.

#### `downloadFile`

```ts
downloadFile(path: string): Promise<Buffer>;
```

Read a file from any path inside the sandbox, returned as a binary Buffer.
Provider mapping: Docker getArchive / Vercel readFileToBuffer / e2b files.read(bytes) / ŌĆ”

#### `uploadFile`

```ts
uploadFile(path: string, content: Buffer): Promise<void>;
```

Write a file (binary) to any path inside the sandbox.
Provider mapping: Docker putArchive / Vercel fs.writeFile(Buffer) / e2b files.write / ŌĆ”

#### `downloadDirectory`

```ts
downloadDirectory(localDir: string, targetDir?: string, opts?: { ignore?: string[] }): Promise<void>;
```

Recursively download an entire directory from inside the sandbox to local disk, symmetric with `uploadDirectory`: writes each file's exact bytes to `localDir` by remote-relative path (creates directories as needed, no text-encoding conversion, no concatenation, no wrapper type with convenience methods). Falls back to workdir (the sandbox-side anchor) when `targetDir` is omitted; `opts.ignore` has the same name and meaning as `uploadDirectory`'s ŌĆö excludes paths by basename, no filtering when omitted.

### `CommandOptions`

#### `env`

```ts
env?: Record<string, string>;
```

Append/override environment variables for this command (merged on top of the sandbox's default environment, not clearing defaults; each provider keeps some of its own fixed variables like `PATH`, which are not guaranteed to be overridable here).

#### `cwd`

```ts
cwd?: string;
```

Working directory for this command; falls back to `Sandbox.workdir` when omitted. Relative paths resolve against workdir, absolute paths are used as-is.

#### `stream`

```ts
stream?: boolean;
```

Also send this command's output into the sandbox's "native log stream" (so `docker logs` / the Docker UI's Logs
tab shows it live). Turn this on for agent commands (codex exec / bub run / claude) and you'll see the agent's raw output
in the container logs. Each provider implements this its own way (docker: tee to the file PID1 tails;
providers that don't support it ignore it) ŌĆö how the log surfaces is the provider's concern, the adapter only declares intent.

#### `onStdout`

```ts
onStdout?: (chunk: string) => void | Promise<void>;
```

Called once for each chunk of the command's stdout as it arrives. The callback is only for short-lived
feedback during the run; the complete stdout still appears as-is in the returned `CommandResult`. When a
provider does not support true streaming, it must at least call this once with the complete stdout after the
command finishes ŌĆö it cannot silently drop it.

#### `onStderr`

```ts
onStderr?: (chunk: string) => void | Promise<void>;
```

The stderr counterpart to `onStdout`; the complete stderr is still retained in `CommandResult`.

#### `root`

```ts
root?: boolean;
```

Run this command as root. Defaults to `false` ŌĆö the command runs as the sandbox's standard **non-root** user (the agent's natural environment).
Use this for installing system dependencies during setup (`apt-get install ŌĆ”`, `pip install --break-system-packages ŌĆ”`).

Semantics stay consistent across providers: "this command runs as root, otherwise as the standard non-root user." Each provider maps this to its own native mechanism
(docker: `exec --user root`; E2B: `{ user: "root" }`; Vercel: `{ sudo: true }`; Daytona: `{ user }`).
Providers that always run as root (e.g. Modal) treat it as a no-op; providers with no privilege escalation at all may not support it (throw) ŌĆö but **the default value and
semantics stay consistent** regardless of provider.

## `ctx` and `t`

`ctx` is the run context seen by the adapter side; `t` is the test context seen by the eval author. They use the same run data but have different responsibilities.
