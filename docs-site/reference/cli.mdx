The [NiceEval](https://niceeval.com/) CLI is the entry point for discovering, running, and viewing evals. Actual eval execution follows an experiment-first model: **`exp` selects a commit-ready run configuration**, and only the positional arguments after the experiment filter eval IDs by prefix. Agent, model, and flags live in `experiments/`, not in ad hoc CLI arguments.

## Output language

The [NiceEval](https://niceeval.com/) CLI and runtime copy support localization without any configuration in `niceeval.config.ts`. When you need a fixed output language, use an environment variable:

```bash
NICEEVAL_LANG=en npx niceeval list
NICEEVAL_LANG=zh-CN npx niceeval list
```

Detection order is `NICEEVAL_LANG`, `NICEEVAL_LOCALE`, `LC_ALL`, `LC_MESSAGES`, then `LANG`. Values starting with `zh` use `zh-CN`; everything else uses `en`. If none are set, the default is `zh-CN`. This only affects terminal/runtime copy — it does not change the machine-readable fields in result JSON, and it does not translate LLM judge prompts.

## `npx niceeval exp [group|config] [id-prefix...]`

```bash
# Run every experiment under experiments/
npx niceeval exp

# Run one experiment group
npx niceeval exp compare-models

# Run only one configuration in that group (id = group/filename, no extension)
npx niceeval exp compare-models/gpt-5.4

# Run every configuration in that group whose filename starts with gpt (e.g. gpt-5.4.ts, gpt-5.4-mini.ts)
npx niceeval exp compare-models/gpt

# Inside that group, only run evals whose ID starts with weather
npx niceeval exp compare-models weather
```

<Note>
  Eval-filter arguments only appear after the experiment selector. Bare `npx niceeval weather` does not run; use `npx niceeval exp local weather` or `npx niceeval exp compare weather` instead.
</Note>

The first positional argument accepts an experiment group name, a single configuration's full id, or a `group/filename-prefix`. An exact match on `group/filename` runs only that one configuration in the group — useful for verifying one configuration at a time or troubleshooting one cell of a comparison. When the filename part doesn't exactly match a full id, it's matched as a prefix instead, running every configuration in the group whose filename starts with it.

## `list`

```bash
npx niceeval list
```

Use this to check that eval discovery, IDs, and config loading are working correctly.

## `exp`

```bash
npx niceeval exp compare-models
npx niceeval exp compare-models weather-tool
```

Run a named experiment, comparing agents, models, or flags in a matrix. Arguments from the second position onward filter eval IDs by prefix.

`show` and `view` are top-level commands — write them as `npx niceeval show` and `npx niceeval view`. If you mistype them as `npx niceeval exp show` / `exp view` and no experiment shares that name, the CLI adds a hint for the correct command after the "no such experiment" error; if the repository does have an experiment with that same name, it still runs as that valid id.

## Two output forms: human text and `--json`

`exp` has exactly two output forms. They only change what gets displayed — never selection, scheduling, verdicts, artifacts, or the exit code:

```bash
# No flag = human text: a dynamic dashboard on a TTY, auto-degrading to an append-only plain stream in pipes/CI logs
npx niceeval exp compare

# --json = the machine face: a single ordered NDJSON event stream on stdout, shared by coding agents, CI adapters, and scripts
npx niceeval exp compare --json

# CI gate: exit code + JUnit; for a JSON summary, redirect `show --json` after the run
npx niceeval exp compare --strict --junit .niceeval/junit.xml
```

See [Runner · Reporter](/explanation/runner#reporter) for how the two forms compare; see [Agent Feedback Loop](/tutorials/agent-feedback-loop) for the full agent loop (reading the event stream, drilling into a locator, targeted reruns); see [CI Integration](/tutorials/ci-integration) for GitHub Actions setup, exit codes, and JUnit.

## `view`

```bash
npx niceeval view
```

Open the local result viewer. It shares the same default report and the same default selection as `show`: for each experiment and each eval, it takes the latest verdict across runs; when you rerun only some evals, the rest are filled in from earlier runs. The default report groups by the experiment id's parent directory and compares only within a group. `show` prints terminal text, `view` renders a web page with interactive evidence browsing; eval ID prefixes, `--exp`, and `--results` narrow both the same way. See [Viewing results](/tutorials/viewing-results) for the full story.

## `show [id-prefix...]`

```bash
npx niceeval show
npx niceeval show weather/brooklyn
npx niceeval show @1k2m9qrs
npx niceeval show @1k2m9qrs --source
npx niceeval show @1k2m9qrs --execution
npx niceeval show fixtures/button --diff
npx niceeval show weather/brooklyn --history
```

`show` is the terminal entry point for results — suited both to a person reading directly and to a coding agent drilling down level by level inside its context window. The positional argument picks which evals to look at (an ID prefix), or picks exactly one attempt with `@<locator>`. With no positional argument, it shows the default comparison report partitioned by experiment group; giving an eval ID prefix only narrows the evals the report covers, it does not change the group boundaries.

`@<locator>` with no evidence flag gives that attempt's compact overview (assertion summary, execution summary, optional OTel timing, diff summary). `--source`, `--execution`, and `--diff` are evidence facets of the same attempt — they expand, respectively, the Eval source captured at run time, the standard execution event stream, and the workspace file changes, so they must be paired with an `@<locator>` that names exactly one attempt. `--results <dir>` sets the results root, and `--history` shows the cross-run trend. See [Viewing results](/tutorials/viewing-results) for the full reading order, output examples, and artifact details.

## `--early-exit` and `--strict`

`--early-exit` defaults to `true`: once one attempt of an eval passes, the runner automatically stops the eval's remaining attempts (to save cost). Passing `--early-exit` explicitly therefore has no effect; what's actually useful is turning it off with `--no-early-exit`, which makes `--runs` > 1 run every attempt to completion (to measure the real pass rate instead of stopping early).

`--strict` does not mean "report errors more strictly" — it changes how soft-threshold assertions are judged. A soft threshold like `.atLeast(n)` (`soft` severity) failing normally does not mark the whole eval as `failed` (it just records an assertion that fell short); with `--strict`, a soft threshold falling short also makes the eval's overall verdict count as `failed`. This is recommended in CI, to avoid the case where "the assertion score is short but the eval shows as passed" slips through.

## Environment variable overrides

The following environment variables correspond to some flags, for cases where passing CLI arguments is inconvenient (such as CI environment configuration):

| Environment variable | Corresponding flag |
|---|---|
| `NICEEVAL_RUNS` | `--runs` |
| `NICEEVAL_MAX_CONCURRENCY` | `--max-concurrency` |
| `NICEEVAL_TIMEOUT` | `--timeout` |
| `NICEEVAL_BUDGET` | `--budget` |

Priority from highest to lowest: CLI flag > environment variable > value in the experiment / `niceeval.config.ts` > built-in default. All four environment variables must be valid numbers; parsing failures exit with an error immediately.

[NiceEval](https://niceeval.com/) also automatically loads a `.env` file from the cwd on startup (no extra configuration needed, and it never overrides an environment variable that already exists) — commonly used to keep credentials like `ANTHROPIC_API_KEY` locally without exporting them manually every time.

## Exit codes

| Exit code | Meaning |
|---|---|
| `0` | Every eval passed (or a command like `list`, `init`, `clean` completed normally). |
| `1` | Some eval is `failed` / `errored`; or a command usage error (such as `exp` matching no experiment, or an unsupported flag combination). |
| `2` | A runtime crash (uncaught exception), caught by the CLI's top-level fallback before exiting. |
| `130` | Exit after receiving an interrupt signal (Ctrl-C / SIGTERM). |

If CI needs soft-threshold failures to also count as failures, combine this with `--strict` (see above).
