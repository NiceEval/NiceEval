# NiceEval

**A progressive, full-featured, developer-friendly, agent-native eval tool for AI agents**

</div>

[NiceEval](https://niceeval.com/) is an agent-native eval tool inspired by [eve](https://eve.dev). It aims for strong developer experience first: a project can usually integrate it in about 10 minutes, eval files stay intuitive, and results stay easy to read.

It is also broad enough to evaluate coding-agent plugins, Hooks, and Skills built for Claude Code / Codex, as well as your own AI agent application. Whether your app is built on AI SDK, LangGraph, Pi, or a custom agent loop, you can connect it through an adapter.

After a run finishes, [NiceEval](https://niceeval.com/) generates readable reports and structured artifacts so you can inspect agent behavior, debug failures, and keep improving pass rate, cost, and latency.

## Architecture

[NiceEval](https://niceeval.com/) supports two integration modes, depending on whether the subject under test needs an isolated workspace.

### Mode 1: Sandbox

Run coding agents such as Codex and Claude Code that need a Sandbox:

```text
   evals/*.eval.ts
        │
        ▼
   ┌────────────┐
   │  niceeval  │
   └────────────┘
        │
        │ Agent adapter (official)
        ▼
   ┌───────────────────────────────┐
   │        Docker Sandbox         │
   │    ┌───────────────────────┐  │
   │    │ Codex / Claude Code / │  │
   │    │   apps that need an   │  │
   │    │  isolated workspace   │  │
   │    └───────────────────────┘  │
   └───────────────────────────────┘
```

### Mode 2: Direct

Connect directly to your own AI agent:

```text
   evals/*.eval.ts
        │
        ▼
   ┌────────────┐
   │  niceeval  │
   └────────────┘
        │
        │ Agent adapter (official / custom)
        ▼
   ┌────────────────────────────┐
   │   Your own AI agent app    │
   │ (AI SDK · LangGraph · Pi · │
   │  custom agent loop, etc.)  │
   └────────────────────────────┘
```

- **[NiceEval](https://niceeval.com/) core** discovers evals, schedules runs, scores them, and generates reports and artifacts.
- **Agent adapters** are the open boundary: you decide how to call the subject under test and how to map its output into the standard event stream.
- **Sandbox** handles file-system isolation: coding agents that need a real workspace use Docker or a cloud Sandbox; your own web agent can connect directly, with no Docker needed.

## Core Concepts

| Concept | In one sentence |
|---|---|
| [Eval](/explanation/evals) | A test case: written in `evals/*.eval.ts`, describing what to test. |
| [Experiment](/explanation/experiment) | A check-in-able run configuration: decides which Adapter, model, and flags to use. |
| [Adapter](/explanation/adapter) | The connection layer to the subject under test: implements a `send` and translates the response into the standard event stream. |
| [Sandbox](/tutorials/sandbox-providers) | Needed only for coding agents that require an isolated workspace; a direct web agent doesn't need one. |
| [Tier](/explanation/tier) | Three investment levels for Adapter integration: Tier 1 only connects `send`, Tier 2 adds OTel for a call waterfall, Tier 3 makes invasive changes for feature A/B tests. |

See [Architecture Overview](/explanation/overview) for the full terminology map.

## Examples

Running one eval needs two files: the eval itself (what to test) and an experiment (which agent to run). The CLI does not accept a bare eval id — in `niceeval exp <experiment> <eval-prefix>`, the experiment is what decides which subject under test gets connected. Below is a real direct-connect web-agent scenario (the full project is at `examples/zh/ai-sdk/` in the repository) that verifies the agent calls a tool for a live weather question and answers from the tool result instead of making one up:

```ts
// evals/weather-tool.eval.ts
import { defineEval } from "niceeval";

export default defineEval({
  description: "Verify that the agent calls the right tool for a live weather question and answers from the result",

  async test(t) {
    const turn = await t.send("What's the weather like in Beijing today?");
    turn.succeeded();

    await t.group("Calls get_weather with the correct city", () => {
      t.calledTool("get_weather", { input: { city: "Beijing" } });
      t.messageIncludes(/°C|temperature|weather|sunny|cloudy|rain/);
    });

    t.judge.autoevals
      .closedQA("Does the assistant answer using the weather data returned by the tool, rather than making up a temperature?")
      .atLeast(0.7);
  },
});
```

```ts
// experiments/local.ts
import { defineExperiment } from "niceeval";
import { webAgent } from "./adapter"; // Your own adapter, connecting to the web agent under test

export default defineExperiment({
  agent: webAgent({ baseUrl: "http://127.0.0.1:5188" }),
  model: "gpt-5.4",
});
```

```sh
npx niceeval exp local weather-tool
npx niceeval show       # read results in the terminal; drill into failures
npx niceeval view       # browse interactively in the web viewer
```

If the subject under test is a coding agent that needs an isolated workspace (Codex, a Claude Code plugin/Skill), use `t.sandbox.uploadDirectory()` in the eval to lay down the workspace, `t.sandbox.fileChanged()` / `t.sandbox.file()` to check which files changed, and `t.sandbox.runCommand()` to run tests — see the standalone repository [coding-agent-skill](https://github.com/CorrectRoadH/coding-agent-skill) for reference.

## FAQ

<AccordionGroup>
  <Accordion title="Do I need to sign up for an account?">
    No. Running evals and viewing reports with `niceeval view` all happen locally. A Sandbox provider such as Docker or E2B is only needed when the subject under test is itself a coding agent that requires an isolated workspace.
  </Accordion>
  <Accordion title="My agent isn't written in TypeScript / JavaScript. Can I still connect it?">
    Yes. An Adapter just sends to and receives from whatever interface your app already exposes — HTTP, gRPC, WebSocket, any of them work. [NiceEval](https://niceeval.com/) doesn't care what language your app is written in or where it's deployed. See [Adapter](/explanation/adapter).
  </Accordion>
  <Accordion title="Can I hook this into CI?">
    Yes. See [CI Integration](/tutorials/ci-integration).
  </Accordion>
</AccordionGroup>
