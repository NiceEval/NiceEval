# NiceEval

**A progressive, full-featured, developer-friendly, agent-native eval tool for AI agents**

</div>

NiceEval is an agent-native eval tool inspired by [eve](https://eve.dev). It aims for strong developer experience first: most projects can integrate in about 10 minutes, eval files stay readable, and results are easy to inspect.

It is also broad enough to evaluate coding-agent plugins, hooks, and Skills built for Claude Code / Codex, as well as your own AI agent application. Whether your app uses AI SDK, LangGraph, Pi, or a custom agent loop, you can connect it through an adapter.

After each run, NiceEval generates readable reports and structured artifacts so you can inspect agent behavior, debug failures, and keep improving pass rate, cost, and latency.

## Architecture

NiceEval supports two integration modes, depending on whether the subject under test needs an isolated workspace.

### Mode 1: Sandbox

Run coding agents such as Codex and Claude Code that need a sandbox:

```text
   evals/*.eval.ts
        |
        v
   ┌────────────┐
   │  niceeval  │
   └────────────┘
        |
        | Adapter (official)
        v
   ┌───────────────────────────────┐
   │        Docker Sandbox         │
   │    ┌───────────────────────┐  │
   │    │ Codex / Claude Code / │  │
   │    │   apps that need an   │  │
   │    │  isolated workspace   │  │
   │    └───────────────────────┘  │
   └───────────────────────────────┘
```

### Mode 2: Direct

Connect directly to your own AI agent:

```text
   evals/*.eval.ts
        |
        v
   ┌────────────┐
   │  niceeval  │
   └────────────┘
        |
        | Adapter (official / custom)
        v
   ┌────────────────────────────┐
   │   Your own AI agent app    │
   │ (AI SDK · LangGraph · Pi · │
   │  custom agent loop, etc.)  │
   └────────────────────────────┘
```

- **NiceEval core** handles eval discovery, scheduling, scoring, reporting, and artifacts.
- **Adapters** are the open boundary: you decide how to call the subject under test and how to map its output into the standard event stream.
- **Sandbox** handles file-system isolation: coding agents that need a real workspace use Docker or a cloud sandbox; direct web agents do not.

## Examples

To run one eval you need two files: the eval itself (what to test) and an experiment (which agent to run). The CLI does not accept a bare eval ID. In `niceeval exp <experiment> <eval-prefix>`, the experiment decides which subject gets connected. Here is a direct web-agent example based on a real scenario:

```ts
// evals/weather-tool.eval.ts
import { defineEval } from "niceeval";

export default defineEval({
  description: "Verify that the agent calls the weather tool and answers from the tool result",

  async test(t) {
    const turn = await t.send("What's the weather like in Beijing today?");
    turn.expectOk();

    await t.group("Calls get_weather with the right city", () => {
      t.calledTool("get_weather", { input: { city: "Beijing" } });
      t.messageIncludes(/°C|temperature|weather|sunny|cloudy|rain/i);
    });

    t.judge.autoevals
      .closedQA("Does the assistant answer from the returned weather data instead of inventing a temperature?")
      .atLeast(0.7);
  },
});
```

```ts
// experiments/local.ts
import { defineExperiment } from "niceeval";
import { webAgent } from "./adapter";

export default defineExperiment({
  agent: webAgent({ baseUrl: "http://127.0.0.1:5188" }),
  model: "gpt-5.4",
});
```

```sh
npx niceeval exp local weather-tool
npx niceeval view
```

If the subject under test is a coding agent that needs an isolated workspace, use `t.sandbox.uploadDirectory()` to lay down the workspace, then verify changed files with `t.sandbox.fileChanged()` / `t.sandbox.file()` and run tests with `t.sandbox.runCommand()`.
