# 用例登记表 —— 先登记，后写测

每个 Feature 测试目录下的 `cases.md` 是该 Feature 契约影响面的登记表（体系定位见[测试体系总览](../testing/README.md)）。本篇定义它的著作权、写测流程和场景预算：登记表是测试数量的闸门，不是写完测试之后的台账。

## 著作权：场景行是设计产物

场景行与 Feature 契约同批定稿。设计迭代重写 `docs/feature/` 受影响小节时，同批写出（或重写）对应 `cases.md` 的场景行——写场景行是设计工作的一部分，交付物是「这条契约由哪些场景证明」，不是测试代码。

这个顺序把测试数量的裁决放到最便宜的评审面上：拒掉一行表格，比删掉一个已经写好的测试函数便宜一个数量级。

## 流程：测试只实现已登记的行

- 实现阶段的测试工作是**把已登记的行变绿**：每条测试对应登记表里的一行，测试名描述该行的契约和场景。
- 实现中途发现的新场景，先补登记行、按契约影响面评审，再写测试。补行是一行 diff，它把「要不要多这条测试」的裁决从代码评审前移到表格评审。
- 写不进登记表的测试不存在：一条测试说不出自己实现哪一行，它就在变更预算之外，按[核心判据](README.md#核心判据)默认删。
- 修 bug 的测试要能追溯台账：bug 修复（先补登记行、先写红测试、再修实现）产生的测试在测试处加一行 `// bug: memory/<条目>.md` 引用对应 memory 条目，登记行注明出处时链同一条目（[sandbox 用例的 Provisioning 分区](sandbox/cases.md)是先例）。memory 是修法台账，复盘「这个修法合理不合理」时要能从测试反查现象与根因，也要能从台账找到锁定它的测试；引用的条目真实存在由仓库守护校验。

## 预算：一行契约一组场景

- 一条契约行登记一组场景：正例、反例、边界各至多一个。要超出这个数，必须证明等价类不同——契约算法与某个常见错误算法在新增输入上给出不同答案（区分力，见 [Harness](harness.md) 的场景规则）；给不出这个区分，新增场景就是同一场景的复述。
- **同一场景的第二条测试是维护负担，不是更多覆盖**——与 [E2E 的仓库 Eval 预算](../e2e-ci/adapters/README.md)同一条纪律。
- 新功能的新增测试上界 = 它新增的登记行。没有新增契约行的改动（实现重构、性能优化）零新增测试是合格状态，不是偷懒；「顺手多补几条」不是勤奋，是给未来每一次契约变更加税。
- 同一不变量要跨故障边界各验证一次（如领域规则层一次、公共组合层一次）时，两个边界各自有登记行，且各行写明自己放走的错误类别——写不出不同的错误类别，就只保留一处。

## 与守护的分工

机器守护（见 [README · 套件边界与仓库守护](README.md#套件边界与仓库守护)）只保证测试文件与登记表不整册脱钩。行级对应靠两条评审义务：测试改动对照登记行核对影响面没有越界；登记行改动对照契约 diff 核对没有超出影响面。
