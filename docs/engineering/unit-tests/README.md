# 单元测试 —— 谁要测、谁不测

这篇定义 niceeval 仓库自己的测试取舍：测试预算往哪儿放、什么代码必须有测试、什么测试不该存在。读者是给这个仓库写测试的人。

跑与验收在 [Build 命令](../../../CLAUDE.md)：`pnpm test` 一条命令跑完全部 vitest（含 docs / memory 一致性守护）。全链路真机验证是另一回事，见 [e2e-ci](../e2e-ci/README.md)。

## 唯一判据

一条测试值不值得存在，只问一句：

> **删掉它，一个真实的 bug 能不能溜进发布？**

答不上来就别写。这条判据推翻了几个常见的写测试理由——「这个函数好测」「这块覆盖率低」「顺手补一个」都不是理由。**测试预算按「静默出错的代价」分配，不按「代码好不好测」分配。**

## 测试预算的分配原则

niceeval 的产品承诺是**「这个判定是可信的」**。据此，同样的 bug 落在不同层，代价差好几个数量级：

| 层 | 出错时 | 用户能发现吗 |
|---|---|---|
| **判定层**（评分、断言、缓存键、调度） | 给出一个**错的答案** | 发现不了。他会拿这个答案去做决策 |
| **归一层**（parser、adapter、normalizer） | 事实缺失或错位，断言静默失灵 | 很难。断言"通过"了，只是它读的事实是空的 |
| **展示层**（report、view、show、CLI 输出） | 画错、排版乱、数字难看 | **一眼看见**。用户会截图发给你 |

所以预算优先级是**判定层 > 归一层 > 展示层**，跟"哪层代码多"「哪层好测」无关。

一个反直觉但必须接受的结论：**展示层测得薄是可以的，判定层测得薄是不可以的**——哪怕前者的代码量是后者的三倍。整体覆盖率是个会骗人的数字：展示层 0.9 / 判定层 0 的套件，比整体均匀 0.5 的套件危险得多。**不追求行覆盖率指标，也不用它验收。**

## 必测：判定链

下列模块的每个导出函数**必须有直接测试**（测试直接 import 它、直接断言它的返回值，不是"通过别的测试顺带跑到"）。它们的共同点是失败时不抛异常，只是给出一个错误答案：

| 模块 | 它错了会发生什么 |
|---|---|
| `src/scoring/verdict.ts` | passed / failed / errored / skipped 的判定本身错了 |
| `src/scoring/` 其余（`collector` `scoped` `match` `judge`） | 断言收集、作用域绑定、匹配、LLM 评委给出错误评分 |
| `src/expect/` | 用户写的断言语义不是他以为的那个 |
| `src/runner/fingerprint.ts` | **缓存键错了 → 把陈旧结果当成刚跑出来的报给用户**。全仓库静默出错风险最高的一处 |
| `src/runner/run.ts`（`runEvals`） | 并发调度、earlyExit 中断去重、budget 记账、carry/resume 合并出错 |
| `src/o11y/derive.ts`（`deriveRunFacts`） | 事实流错了 → **每一条**基于事实的用户断言跟着错 |

写这些测试不需要真沙箱、真 agent：`src/runner/attempt.test.ts` 里的 `FakeSandbox` 是已经验证过的驱动方式，`fingerprint` / `verdict` 是纯函数，直接喂输入断输出。

## 该测，但可以薄：归一层

parser（`src/o11y/parsers/`）、adapter（`src/agents/`）、normalizer 的测试形态是固定的：

> **喂一份真实形状的 payload，断言归一化后的输出。**

判据是 payload 的**来源**——从真实 SDK / CLI 输出录下来的，测了才算数；凭印象手搓一个 fake 事件再断言它原样转一圈，测的是你自己的想象。（`src/agents/sdk-streams.test.ts` 是这类测试的样板：145 行锁住 462 行的映射，含一个真实 CI 事故的回归锁。）

每个 parser / adapter 至少要有一份这样的测试。它不需要大——薄而真 > 厚而假。

## 不写的测试

下面这些**明确不要写**，见到了就删。它们的共同点是：能失败，但失败时不代表任何 bug。

- **改名检测器 / 拷贝比对**。把一份源码复制到 `test/fixtures/` 再比对两者输出——它唯一能失败的方式是有人改了一个忘了改另一个，抓不到任何 bug，只在每次重构时收改名税。要证明「公开 API 足够用户重建 X」，让 fixture **能编译过**就已经证完了，不需要输出比对。
- **grep 源码文本的测试**。`readFileSync(某个源文件)` 然后正则匹配它的内容——那是 lint 规则穿了测试的马甲。改个局部变量名就红，且它保护的行为通常在别处已有真实的行为测试。
- **断言三方库能工作**。哈希函数不撞车、SDK 的构造函数能构造、`JSON.parse` 会解析——这些不是本仓库的契约。
- **测没有生产调用者的导出**。先决定这个函数该不该存在：要么删掉它，要么让生产代码用它。给一个死函数配测试，是在给死代码续命。
- **断言一个不存在的字段不存在**。`expect(x).not.toHaveProperty("已删除的字段")` —— 类型系统已经管了这件事。
- **把类型断言当测试**。整个文件全是 `@ts-expect-error`、运行时零断言——它属于 `pnpm run typecheck`，不属于 `pnpm test`。
- **重复断言同一件事**。同一个不变量在第三个文件里被断言时，前两个里至少有一个该删。同一份 fixture 的同一个渲染面，只在**一个**文件里断言。

## 套件边界

vitest 只收**本仓库自己的**测试。`vitest.config.ts` 的 `exclude` 必须排除：

- `.repos/**` —— vendored 的外部仓库。
- `.claude/**` —— agent 的临时 worktree，每个里面含一整份 `src/` 副本。

后者尤其要紧：worktree 里的副本会被当成正式测试跑，但它们跑的是**该 worktree 那一刻的旧源码**。这种测试既不可能抓到当前代码的回归，又能因为与本次改动完全无关的陈旧原因把 CI 弄红。出处见 [`vitest-collects-agent-worktree-copies`](../../../memory/vitest-collects-agent-worktree-copies.md)。

判据：`pnpm test` 报出的文件数，应当等于 `src/` 和 `test/` 下测试文件的实际数量。对不上就是收进了不该收的东西。

## 守护类测试

仓库约定需要机器校验时（索引覆盖、链接真实性、生成区块漂移），写成 `test/` 下的 vitest 测试，**不新增脚本、package.json 命令或 hook**。这条规则和它已批准的例外，定义在 [CLAUDE.md](../../../CLAUDE.md)。

这类测试不受「唯一判据」约束——它们守护的不是代码 bug，而是仓库约定的腐坏（一条死链、一个漏索引的 memory 条目、一份漂移的生成文档）。这类腐坏没有别的机制会发现。
